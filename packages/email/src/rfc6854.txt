       <span class="h1">Update to Internet Message Format to Allow Group Syntax in</span>
                <span class="h1">the &quot;From:&quot; and &quot;Sender:&quot; Header Fields</span>

Abstract

   The Internet Message Format (<a href="/doc/html/rfc5322">RFC 5322</a>) allows &quot;group&quot; syntax in some
   email header fields, such as &quot;To:&quot; and &quot;CC:&quot;, but not in &quot;From:&quot; or
   &quot;Sender:&quot;.  This document updates <a href="/doc/html/rfc5322">RFC 5322</a> to relax that restriction,
   allowing group syntax in those latter fields, as well as in
   &quot;Resent-From:&quot; and &quot;Resent-Sender:&quot;, in certain situations.

<span class="h2"><a class="selflink" id="section-1" href="#section-1">1</a>.  Introduction</span>

   The Internet Message Format, as far back as <a href="/doc/html/rfc822">RFC 822</a> [<a href="/doc/html/rfc0822" title="&quot;Standard for the format of ARPA Internet text messages&quot;">RFC0822</a>], has
   always required a usable address to appear in the &quot;From:&quot; header
   field of messages in order to allow replies to be sent.  To this end,
   the syntax of messages, up to and including the current specification
   [<a href="/doc/html/rfc5322" title="&quot;Internet Message Format&quot;">RFC5322</a>], has required the use of the mailbox address form in the
   originator (&quot;From:&quot; and &quot;Sender:&quot;) fields of messages and has
   specifically forbidden the use of the group address form, which
   permits an empty list of addresses (that is, an address list with no
   address included that might be used for a reply).

   However, the use cases for the &quot;From:&quot; field have evolved.  There are
   numerous instances of automated systems that wish to send email but
   cannot handle replies, and a &quot;From:&quot; field with no usable addresses
   would be extremely useful for that purpose.  More recently, work with
   internationalized email addresses [<a href="/doc/html/rfc6530" title="&quot;Overview and Framework for Internationalized Email&quot;">RFC6530</a>] creates a real need to
   take a message with an internationalized email address and hand it to
   an older client that would have no ability to reply to such an
   address but might still wish to display the contents of the message.
   The group construct provides an existing syntax for unusable
   addresses (using the empty list of addresses) and also allows for a
   text label that describes the originator.  For example:

      From: Automated System:;

   A review of many current email programs finds that all reviewed
   clients will properly display a message with group syntax in the
   &quot;From:&quot; field.  At worst, such programs generate an error message
   when an attempt is made to reply to such a message.  No other
   interoperability problems have been discovered.



<span class="grey">Leiba                        Standards Track                    [Page 2]</span></pre>
<hr class='noprint'/><!--NewPage--><pre class='newpage'><span id="page-3" ></span>
<span class="grey"><a href="/doc/html/rfc6854">RFC 6854</a>          Group Syntax in &quot;From:&quot; and &quot;Sender:&quot;       March 2013</span>


   This document therefore updates the Internet Message Format
   specification [<a href="/doc/html/rfc5322" title="&quot;Internet Message Format&quot;">RFC5322</a>] to relax that restriction, allowing group
   syntax to be used in the originator (&quot;From:&quot; and &quot;Sender:&quot;) fields,
   as well as in their corresponding resent (&quot;Resent-From:&quot; and
   &quot;Resent-Sender:&quot;) fields.  This change permits empty groups, as
   described above, and also permits named groups of mailboxes (groups
   with non-empty lists of addresses; see <a href="#section-4">Section 4</a>).  Nevertheless,
   this document recommends against the general use of group syntax in
   these fields at this time (see <a href="#section-3">Section 3</a>).

<span class="h3"><a class="selflink" id="section-1.1" href="#section-1.1">1.1</a>.  Notational Conventions</span>

   The notational conventions here are the same as those in <a href="/doc/html/rfc5322">RFC 5322</a>,
   and the following two subsections are copied directly from that
   document.

<span class="h4"><a class="selflink" id="section-1.1.1" href="#section-1.1.1">1.1.1</a>.  Requirements Notation</span>

   This document occasionally uses terms that appear in capital letters.
   When the terms &quot;MUST&quot;, &quot;SHOULD&quot;, &quot;RECOMMENDED&quot;, &quot;MUST NOT&quot;, &quot;SHOULD
   NOT&quot;, and &quot;MAY&quot; appear capitalized, they are being used to indicate
   particular requirements of this specification.  A discussion of the
   meanings of these terms appears in the Key Words document [<a href="/doc/html/rfc2119" title="&quot;Key words for use in RFCs to Indicate Requirement Levels&quot;">RFC2119</a>].

<span class="h4"><a class="selflink" id="section-1.1.2" href="#section-1.1.2">1.1.2</a>.  Syntactic Notation</span>

   This specification uses the Augmented Backus-Naur Form (ABNF)
   [<a href="/doc/html/rfc5234" title="&quot;Augmented BNF for Syntax Specifications: ABNF&quot;">RFC5234</a>] notation for the formal definitions of the syntax of
   messages.  Characters will be specified either by a decimal value
   (e.g., the value %d65 for uppercase A and %d97 for lowercase A) or by
   a case-insensitive literal value enclosed in quotation marks (e.g.,
   &quot;A&quot; for either uppercase or lowercase A).

<span class="h2"><a class="selflink" id="section-2" href="#section-2">2</a>.  Allowing Group Syntax in &quot;From:&quot; and &quot;Sender:&quot;</span>

   <a href="/doc/html/rfc5322#section-3.6.2">Section&nbsp;3.6.2 of RFC 5322</a> defines the &quot;From:&quot; header field as
   containing a &lt;mailbox-list&gt; syntax element.  This specification
   changes that definition to use the &lt;address-list&gt; syntax element, as
   is used in other fields, such as &quot;To:&quot;, &quot;CC:&quot;, and &quot;Reply-To:&quot;.  This
   specification also changes the definition of the &quot;Sender:&quot; header
   field from the &lt;mailbox&gt; syntax element to the &lt;address&gt; syntax
   element.  While the &lt;address&gt; element includes the &lt;mailbox&gt; element
   already, we have chosen to specify both in the updated syntax as a
   way of highlighting the limited use intended for the change (see
   <a href="#section-3">Section 3</a>).

<hr class='noprint'/><!--NewPage--><pre class='newpage'><span id="page-4" ></span>

   <a href="#section-2.1">Section 2.1</a> below is a full replacement for Section 3.6.2 of <a href="/doc/html/rfc5322">RFC</a>
   <a href="/doc/html/rfc5322">5322</a>, containing the new syntax as well as a new description of the
   semantics for the &quot;From:&quot; and &quot;Sender:&quot; fields.  <a href="#section-2.2">Section 2.2</a> below is
   a replacement of only the ABNF syntax for the &quot;Resent-From:&quot; and
   &quot;Resent-Sender:&quot; fields in <a href="/doc/html/rfc5322#section-3.6.6">Section&nbsp;3.6.6 of RFC 5322</a>; the rest of the
   syntax as well as the descriptive text of <a href="/doc/html/rfc5322#section-3.6.6">Section&nbsp;3.6.6 of RFC 5322</a>
   remains unchanged.

   [The text in the following section is not consistent within itself

<span class="h3"><a class="selflink" id="section-2.1" href="#section-2.1">2.1</a>.  Replacement of <a href="/doc/html/rfc5322#section-3.6.2">RFC 5322, Section&nbsp;3.6.2</a>. Originator Fields</span>

   The originator fields of a message consist of the from field, the
   sender field (when applicable), and optionally the reply-to field.
   The from field consists of the field name &quot;From&quot; and a
   comma-separated list of one or more addresses (either mailbox or
   group syntax).  If the from field contains more than one mailbox
   specification (including all mailboxes included in any groups), then
   the sender field, containing the field name &quot;Sender&quot; and a single
   address, MUST appear in the message.  The from field and the sender
   field SHOULD NOT use group syntax; rather, the from field SHOULD use
   only the mailbox-list syntax and the sender field SHOULD use only
   mailbox syntax (see <a href="/doc/html/rfc6854#section-3">RFC 6854, Section&nbsp;3</a>).  If the sender field uses
   group syntax, the group MUST NOT contain more than one mailbox.  In
   either case, an optional reply-to field MAY also be included, which
   contains the field name &quot;Reply-To&quot; and a comma-separated list of one
   or more addresses.

   from = &quot;From:&quot; (mailbox-list / address-list) CRLF

   sender = &quot;Sender:&quot; (mailbox / address) CRLF

   reply-to = &quot;Reply-To:&quot; address-list CRLF

   The originator fields indicate the mailbox(es) of the source of the
   message.  The &quot;From:&quot; field specifies the author(s) of the message,
   that is, the mailbox(es) of the person(s) or system(s) responsible
   for the writing of the message.  The &quot;Sender:&quot; field specifies the
   mailbox of the agent responsible for the actual transmission of the
   message.  For example, if a secretary were to send a message for



<span class="grey">Leiba                        Standards Track                    [Page 4]</span></pre>
<hr class='noprint'/><!--NewPage--><pre class='newpage'><span id="page-5" ></span>
<span class="grey"><a href="/doc/html/rfc6854">RFC 6854</a>          Group Syntax in &quot;From:&quot; and &quot;Sender:&quot;       March 2013</span>


   another person, the mailbox of the secretary would appear in the
   &quot;Sender:&quot; field and the mailbox of the actual author would appear in
   the &quot;From:&quot; field.  If the originator of the message can be indicated
   by a single mailbox and the author and transmitter are identical, the
   &quot;Sender:&quot; field SHOULD NOT be used.  Otherwise, both fields SHOULD
   appear.

      Note: The transmitter information is always present.  The absence
      of the &quot;Sender:&quot; field is sometimes mistakenly taken to mean that
      the agent responsible for transmission of the message has not been
      specified.  This absence merely means that the transmitter is
      identical to the author and is therefore not redundantly placed
      into the &quot;Sender:&quot; field.

   The originator fields also provide the information required when
   replying to a message.  When the &quot;Reply-To:&quot; field is present, it
   indicates the address(es) to which the author of the message suggests
   that replies be sent.  In the absence of the &quot;Reply-To:&quot; field,
   replies SHOULD by default be sent to the mailbox(es) specified in the
   &quot;From:&quot; field unless otherwise specified by the person composing the
   reply.

   In all cases, the &quot;From:&quot; field SHOULD NOT contain any mailbox that
   does not belong to the author(s) of the message.  See also <a href="/doc/html/rfc5322#section-3.6.3">Section</a>
   <a href="/doc/html/rfc5322#section-3.6.3">3.6.3 of RFC 5322</a> [<a href="/doc/html/rfc5322" title="&quot;Internet Message Format&quot;">RFC5322</a>] for more information on forming the
   destination addresses for a reply.

<span class="h3"><a class="selflink" id="section-2.2" href="#section-2.2">2.2</a>.  Update to <a href="/doc/html/rfc5322#section-3.6.6">RFC 5322, Section&nbsp;3.6.6</a>. Resent Fields</span>

   This section updates <a href="/doc/html/rfc5322#section-3.6.6">RFC 5322, Section&nbsp;3.6.6</a>, to allow groups (via
   the address-list ABNF production) in the &quot;Resent-From:&quot; and
   &quot;Resent-Sender:&quot; fields, to parallel the change to &quot;From:&quot; and
   &quot;Sender:&quot; above.  The ABNF for these fields is changed as follows:

   resent-from = &quot;Resent-From:&quot; (mailbox-list / address-list) CRLF

   resent-sender = &quot;Resent-Sender:&quot; (mailbox / address) CRLF

<span class="h2"><a class="selflink" id="section-3" href="#section-3">3</a>.  Applicability Statement</span>

   Mailbox syntax is the normal syntax to use in the &quot;From:&quot; and
   &quot;Sender:&quot; header fields; the address syntax defined in <a href="#section-2.1">Section 2.1</a>,
   which allows the specification of a group, is only for Limited Use
   (see <a href="/doc/html/rfc2026">RFC 2026</a> <a href="/doc/html/rfc2026#section-3.3">[RFC2026], Section&nbsp;3.3</a>, item (d)) for the reasons
   described below.

   Many Internet email procedures and much software assumes that the
   addresses in the &quot;From:&quot; and &quot;Sender:&quot; fields can be replied to and
   are suitable for use in organizing and filtering mail.  The use of
   groups instead of mailboxes can disrupt these uses.  Consequently,
   while this specification legitimizes the use of groups, it does so
   only to enable circumstances when that use is necessary.  Because the
   use of this mechanism is new, it is important that its use be limited
   to these circumstances and that it be used with caution.  In
   particular, user agents SHOULD NOT permit the use of groups in those
   fields in outgoing messages.

<span class="h2"><a class="selflink" id="section-4" href="#section-4">4</a>.  Examples</span>

   First, consider an email message that is sent by an automated nightly
   monitor program, to which replies should not be sent.  Such messages
   commonly include a valid, replyable address that will discard any
   replies that are sent to it, but recipients who do reply might be
   unaware that their replies will be discarded.  If the message is
   instead presented as follows, the recipients&#x27; email clients will not
   allow them to reply in the first place:

      From: Nightly Monitor Robot:;

   Second, consider an email message that is meant to be &quot;from&quot; the two
   managing partners of a business, Ben and Carol, and that is sent by
   their assistant, Dave.  This message could always have been presented
   this way:

      From: ben@example.com,carol@example.com
      Sender: dave@example.com

   This change allows it to be represented this way:

      From: Managing Partners:ben@example.com,carol@example.com;
      Sender: dave@example.com

<span class="h2"><a class="selflink" id="section-5" href="#section-5">5</a>.  Security Considerations</span>

   See the Internet Message Format specification [<a href="/doc/html/rfc5322" title="&quot;Internet Message Format&quot;">RFC5322</a>] for general
   discussion of security considerations related to the formatting of
   email messages.

   The &quot;From:&quot; address is special, in that most user agents display this
   address, or the &quot;friendly&quot; text associated with it, to the end user,
   and label it so as to identify it as the origin of the message (as
   implied in <a href="/doc/html/rfc5322#section-3.6.2">Section&nbsp;3.6.2 of RFC 5322</a>).  Group syntax in the &quot;From:&quot;
   header field can be used to hide the identity of the message
   originator.  It is just as easy to use a fabricated &quot;From:&quot; address
   to accomplish the same thing, so allowing groups in this field does
   not exacerbate the security problem.

   Some protocols attempt to validate the originator address by matching
   the &quot;From:&quot; address to a particular verified domain (for one such
   protocol, see the Author Domain Signing Practices (ADSP) document
   [<a href="/doc/html/rfc5617" title="&quot;DomainKeys Identified Mail (DKIM) Author Domain Signing Practices (ADSP)&quot;">RFC5617</a>]).  Such protocols will not be applicable to messages that
   lack an actual email address (whether real or fake) in the &quot;From:&quot;
   field.  Local policy will determine how such messages are handled,
   and senders, therefore, need to be aware that using groups in the
   &quot;From:&quot; might adversely affect deliverability of the message.

   Because groups have previously not been allowed in the &quot;From:&quot; and
   &quot;Sender:&quot; header fields, it is possible that some implementations
   that conform to <a href="/doc/html/rfc5322">RFC 5322</a> might not be prepared to handle the group
   syntax, and, indeed, might not even recognize that group syntax is
   being used.  Of those implementations, some subset might, when
   presented with group syntax in those header fields, behave in a way
   that is exploitable by an attacker.  It is deemed unlikely that this
   will be a serious problem in practice: address field parsing is
   generally an integral component of implementations, and address field
   parsers are required to understand group syntax.  In addition, if any
   implementations should be exploitable through this mechanism, it is
   already possible for attackers to do it by violating <a href="/doc/html/rfc5322">RFC 5322</a>.  Other
   violations of <a href="/doc/html/rfc5322">RFC 5322</a> are commonly used by malefactors.
