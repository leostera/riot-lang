## ⏸️ Deferred Features

### Binary/Executable Compilation
**Status:** Intentionally deferred

**Requires:**
- Package dependency graph
- Topologically sorted package dependencies
- Cross-package `.cmxa` resolution
- Unix library detection across dependency chain
- Custom executable creation

**Reason:** These require context from the broader build system that the planner doesn't (and shouldn't) have access to.

**Solution:** Handle in main tusk orchestration layer which has full build graph.

## 🚫 Not Needed

### CopyDir Action
**Status:** Not needed in planner

**Reason:** `CopyDir` is used in sandbox setup (before planning), not during module compilation. It's handled by the `Sandbox` module in main tusk.

### GenerateInterface Action  
**Status:** Defined but not generated

**Reason:** Never generated by main tusk either. Likely an old/unused action type.

## 🎯 What the Planner Does

The planner is responsible for **single-package library compilation**:

1. Scan package source directory
2. Build module dependency graph
3. Generate compilation actions
4. Create library archive
5. Return actions + outputs for execution

## 🚫 What the Planner Does NOT Do

The planner is **not** responsible for:

1. Multi-package builds (that's the build graph)
2. Executable linking (requires cross-package context)
3. Sandbox creation (that's the sandbox module)
4. Action execution (that's the executor)
5. Caching (that's the store)
6. Build orchestration (that's the build planner)

## ✨ Design Philosophy

The planner is a **focused, single-responsibility module**:

- Input: Package info + source directory
- Output: Module graph + action graph + outputs
- No global state
- No side effects (except ocamldep calls)
- Pure planning, no execution

This clean separation makes it:
- Testable
- Composable
- Maintainable
- Ready for integration
