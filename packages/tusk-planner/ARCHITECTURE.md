# tusk-planner Architecture

## Overview

The planner package orchestrates the complete build planning process for OCaml packages, from source scanning to action graph generation.

## Modules

### Core Planning

#### `Planner`
High-level planning orchestrator.

**API:**
```ocaml
type plan_input = {
  package : Package.t;
  toolchain : Toolchains.toolchain;
  workspace : Workspace.t;
  source_dir : Path.t;
  sandbox_dir : Path.t;
}

type plan_result =
  | Planned of {
      module_graph : Module_node.t Graph.SimpleGraph.t;
      action_graph : Action_graph.t;
      outputs : Path.t list;
    }
  | Cycle of { cycle : string list }
  | Error of string

val plan_node : plan_input -> plan_result
```

**Process:**
1. Creates module dependency graph from sources
2. Wires dependencies via ocamldep
3. Topologically sorts modules
4. Generates action graph
5. Returns ready-to-execute plan

---

### Module Graph Construction

#### `Graph_builder`
Builds module dependency graph in two phases.

**Phase 1 - Structural Scanning:**
- Scans source directory recursively
- Creates nodes for ML/MLI/C/H files
- Generates library interface modules
- Creates alias modules for namespace flattening
- Adds structural edges (parent-child, ML→MLI, library deps)

**Phase 2 - Dependency Wiring:**
- Collects concrete ML/MLI files
- Runs `Ocamldep.batch_deps` (single call for all files)
- Adds module reference edges
- Filters MLI → ML edges

**Key Functions:**
```ocaml
val create : config -> t
val wire_dependencies : t -> Path.t -> unit
val graph : t -> Module_node.t Graph.SimpleGraph.t
```

#### `Module_scanner`
Directory traversal and file discovery.

- Pre-sorts files (MLI before ML, directories last)
- Returns tagged variants for type safety
- Handles nested directory structures

#### `Module_registry`
Maps module names to graph node IDs for dependency lookup.

---

### Graph Nodes

#### `Module_node`
Represents compilable units in the module graph.

**Node Types:**
- `ML` - Implementation file
- `MLI` - Interface file  
- `C` - C source file
- `H` - Header file
- `Root` - Graph root

**File Sources:**
- `Concrete of Path.t` - Actual source file
- `Generated of { path; contents }` - Generated code

---

### Library Support

#### `Library_definition`
Analyzes directory structure to determine library composition.

**Responsibilities:**
- Identifies child files vs subdirectories
- Detects concrete vs generated interface
- Filters binaries
- Determines library interface dependencies

#### `Library_interface`
Generates library interface modules (e.g., `Foo.ml` for `foo/` directory).

**Generated Content:**
```ocaml
(* Library interface module generated by tusk *)
module Bar = Foo__Bar
module Baz = Foo__Baz
```

#### `Alias_module`
Generates alias modules for namespace flattening.

**Purpose:** All files in a library implicitly `open Foo__Aliases` to access sibling modules without qualification.

---

### Action Graph

#### `Action_graph`
Dependency graph of build actions for parallel execution.

**Action Types:**
- `CompileInterface` - Compile .mli → .cmi
- `CompileImplementation` - Compile .ml → .cmo
- `GenerateInterface` - Generate .mli from .ml (ocamlc -i)
- `CompileC` - Compile C source
- `CreateLibrary` - Link .cma
- `CreateExecutable` - Link executable
- `CopyFile` - File operations
- `WriteFile` - Generate files

**Graph Structure:**
```ocaml
type action_node = {
  action : action;
  outputs : Path.t list;
}
```

**Operations:**
```ocaml
val create : unit -> t
val add_action : t -> action -> outputs:Path.t list -> action_node G.node
val add_dependency : t -> action_node G.node -> depends_on:action_node G.node -> unit
val topo_sort : t -> action_node G.node list
```

## Data Flow

```
Source Files
    ↓
Module_scanner.scan
    ↓
Graph_builder.create
    ↓ (Phase 1: Structural)
Module Graph (structural edges only)
    ↓
Graph_builder.wire_dependencies
    ↓ (Phase 2: via ocamldep)
Module Graph (complete)
    ↓
Generate Actions (TODO)
    ↓
Action Graph
    ↓
Planner.plan_node
    ↓
plan_result (ready for executor)
```

## Integration Example

```ocaml
open Tusk_planner

let plan_build ~package ~toolchain ~workspace ~source_dir ~sandbox_dir =
  let input = Planner.{
    package;
    toolchain;
    workspace;
    source_dir;
    sandbox_dir;
  } in
  
  match Planner.plan_node input with
  | Planned { module_graph; action_graph; outputs } ->
      (* Execute action_graph with worker pool *)
      Ok (action_graph, outputs)
  | Cycle { cycle } ->
      Error (format "Circular dependency: %s" (String.concat " -> " cycle))
  | Error msg ->
      Error msg
```

```ocaml

  match plan with
  | Planned { action_graph; _ } ->
      (* tusk-executor will consume this *)
      let sorted_actions = Action_graph.topo_sort action_graph in
      Worker_pool.execute_parallel sorted_actions
```

## Design Principles

1. **Two-Phase Construction** - Separate structural scanning from dependency wiring for clarity
2. **Batch Operations** - Single ocamldep call instead of per-file for performance
3. **Type Safety** - Abstract types and tagged variants throughout
4. **No Magic** - Explicit algorithms, documented edge cases
5. **Clean Boundaries** - No circular dependencies, clear module responsibilities
