Riot is an OCaml toolchain that tries to make one workflow feel coherent:

- create or grow a workspace
- build packages predictably
- run binaries through the same package model
- test and benchmark them in one system
- generate docs from the public interface
- publish the result when it is ready

This site is written for the four commitment levels most users actually have:

- `browsing`: what does Riot cover?
- `evaluating`: would this fit my team and workflow?
- `trying`: can I get one real package working quickly?
- `depending`: what are the stable loops I will use every day?

That is why the main tabs are:

- [Toolchain](/toolchain/index)
- [Package Manager](/package-manager/index)
- [Test Runner](/test-runner/index)

They correspond to the three biggest jobs Riot is trying to solve.

## If you are new to Riot

Start with this path:

1. [Installation](/toolchain/installation)
2. [Quick Start](/toolchain/quickstart)
3. [riot build](/toolchain/build)
4. [riot run](/toolchain/riot-run)
5. [riot test](/test-runner/riot-test)
6. [riot doc](/package-manager/doc)

That path exercises the product end to end. It proves Riot is not only a binary on your machine, but a
usable workflow.

## What Riot is trying to do differently

Riot is not trying to be a random pile of useful commands.

It is trying to make these pieces reinforce each other:

- package structure
- dependency management
- build planning and execution
- runtime-oriented libraries
- JSON and docs output for tooling

The docs are organized the same way. You should be able to stay inside one tab for a while and still
feel like the system is connected.
