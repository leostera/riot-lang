## Summary
[summary]: #summary

This RFD defines Riot's final low-level I/O model around owned off-heap
buffers, borrowed byte slices, and one explicit buffered-reader layer. The goal
is not to make every public API borrowed. The goal is to make the fast
transport substrate the default inside `kernel` and `std`, while keeping
ownership changes explicit and leaving public protocol APIs ergonomic where the
benchmark evidence says that still matters.

- `Kernel.IO` owns the syscall-facing byte model: `Buffer`, `IoSlice`, and
  `IoVec`
- `Std.IO` re-exports that byte model and uses a single public `IO.Error` and
  `IO.result`
- `Std.IO.Reader` and `Std.IO.Writer` are buffer-first; `Std.IO.BufReader` is
  the only borrowed-slice layer in `std`
- `Std.IO.Buffer` is the default off-heap buffer in `std`; heap text
  accumulation lives on `Std.StringBuilder`
- out of scope: a generic incremental parser framework, borrowed HTTP request
  and response APIs by default, or hiding copy boundaries behind convenience
  helpers

## Motivation
[motivation]: #motivation

Riot had two structural problems in its I/O stack.

First, the transport substrate was not explicit enough. Heap `string`, heap
`bytes`, and off-heap syscall buffers kept being treated as if they were
interchangeable. They are not. Heap values are convenient, but they are the
wrong thing to carry through the blocking syscall boundary. Off-heap storage is
stable there, but until this redesign it was not the obvious default in
`Std.IO`.

Second, the standard-library API shape was teaching the wrong habits. `Std.IO`
had a `Buffer` that behaved like a heap string builder while the actual off-heap
buffer sat behind a more specific name. Reader and writer APIs mixed transport
helpers with string-owning helpers, and public type parameters exposed source
and error details that callers did not need. The result was predictable:

- hot paths were easy to route through heap strings by accident
- copy boundaries were hard to see in code review
- composition points such as `Reader.chain`, `Reader.take`, and `BufReader.reset`
  carried more type noise than actual signal

The HTTP work made the trade-off concrete.

Parser-only request parsing on large bodies clearly benefits from staying on
`IoSlice` until the final ownership boundary:

- parser-only `10 MiB` request:
  - baseline string parser: `6.24 ms`
  - public `parse`: `308.80 us`
  - `parse_slice`: `12.60 us`
  - `Borrowed.parse`: `8.40 us`

Full-request reader-driven parsing also benefits once ingestion uses caller-owned
off-heap buffers:

- reader-driven `10 MiB` request:
  - baseline string parser: `23.71 ms`
  - public `parse`: `15.04 ms`
  - `parse_slice`: `2.79 ms`
  - `Borrowed.parse`: `2.38 ms`

But the same benchmarks also show where *not* to overreach:

- reader-driven `small request`:
  - baseline string parser: `4.51 us`
  - `parse_slice`: `10.02 us`
- reader-driven GitHub-like request:
  - baseline string parser: `97.85 us`
  - `parse_slice`: `179.38 us`

That matters. Riot should not force borrowed slices all the way up every public
protocol API when the gains are concentrated in the transport substrate and in
large-body parsing. The right move is narrower:

- make the transport substrate off-heap by default
- make copy boundaries explicit
- keep borrowed slice use explicit and local to the layers where it pays
- keep public protocol values owned where that is still the right ergonomic
  trade-off

The same pattern shows up in the `std` benches. `BufReader.read_line` and
`BufReader.read_slice |> IoSlice.to_string` are within a few percent of each
other on larger line sizes, while borrowed slices pull ahead mostly when they
avoid repeated tiny materializations. So the substrate matters, but the public
story should stay honest about where ownership changes still belong.

If Riot does nothing, we should expect the same problems to recur:

- more silent drift back toward heap buffers in hot paths
- more one-off fixes around blocking native I/O and buffer ownership
- more transport code that looks ergonomic but hides avoidable copies
- more friction when composing readers, writers, and buffered readers

## Guide-level explanation
[guide-level-explanation]: #guide-level-explanation

The intended mental model is the same split that high-performance systems in Go
and Rust converge on:

- raw readers fill caller-owned buffers
- buffered readers may hand out borrowed slices from their internal buffer

For Riot that means:

- `Std.IO.Buffer` owns contiguous off-heap bytes
- `Std.IO.IoSlice` is a borrowed byte view
- `Std.IO.IoVec` is a vectored collection of slices
- `Std.IO.Reader` fills caller-owned buffers
- `Std.IO.BufReader` owns one reusable internal buffer and borrows slices from it
- `Std.IO.Writer` drains caller-owned buffers
- `Std.StringBuilder` is for explicit heap text accumulation, not transport

### Workflow 1: transport-first parsing

Suppose a caller is reading a request from a socket and wants to parse it
without bouncing through a heap string first.

With this model the normal flow is:

```ocaml
let buf = IO.Buffer.create ~size:4096 in
let* _ = IO.Reader.read reader ~into:buf in

let slice = IO.Buffer.readable buf in
match Http1.Request.parse_slice slice with
| Done { value; remaining } -> ...
| Need_more -> ...
| Error error -> ...
```

The important thing here is what does *not* happen:

- `Reader.read` does not allocate and return a hidden borrowed slice
- `parse_slice` does not need a whole-input `string -> IoSlice` adapter
- the caller sees exactly where transport bytes become owned text

### Workflow 2: borrowed scanning over a buffered reader

Suppose a caller is doing line-oriented or delimiter-oriented parsing.

That is what `BufReader` is for:

```ocaml
let br = IO.BufReader.from_reader reader in
let* line = IO.BufReader.read_line br in
let owned = IO.IoSlice.to_string line in
...
```

Here the borrowed-lifetime rule is the main teaching point:

- the returned `IoSlice` is borrowed from `BufReader`'s internal buffer
- it remains valid until the next operation that may refill, consume, or reset
  that buffer
- if the caller wants to keep it longer, `IoSlice.to_string` or `IoSlice.to_bytes`
  is the explicit ownership boundary

This makes the code review story better. A reviewer can see:

- where transport bytes are stored
- whether a parser is staying on slices or materializing strings
- whether a borrowed value can outlive the buffer it came from

### What contributors should assume

- inside `Std.IO`, the default currency is `Buffer`, `IoSlice`, and `IoVec`
- `from_*` and `to_*` are explicit copy boundaries
- raw `Reader` and `Writer` APIs are about caller-owned buffers
- `BufReader` is the only standard-library surface that returns borrowed slices
- public protocol APIs may still expose owned strings where that remains the
  right default

### What contributors should not assume

- that heap `bytes` is the right default transport buffer
- that every parser should become borrowed by default
- that zero-copy means Riot never copies
- that borrowed slices should leak upward just because they are faster in one
  benchmark shape

## Reference-level explanation
[reference-level-explanation]: #reference-level-explanation

## 1. Kernel byte model

### 1.1 `Kernel.IO.IoSlice`

`IoSlice` is the basic borrowed byte view over off-heap storage.

It should:

- support zero-copy sub-slicing
- keep delimiter and text-ish helpers directly on the slice
- treat `from_*` and `to_*` as explicit copy boundaries
- return `result` for range-sensitive operations
- keep `_unchecked` variants for inner loops that have already validated bounds

`IoSlice` replaces the need for a separate `StringView` type in the core model.
The byte view is enough; the important distinction is ownership, not a second
text-only wrapper.

### 1.2 `Kernel.IO.IoVec`

`IoVec` is the narrow vectored-I/O type for `readv` and `writev`.

It should stay:

- syscall-facing
- cheap to build from slices
- explicit about its relationship to `IoSlice`

It is not a string builder and not a parser abstraction.

### 1.3 `Kernel.IO.Buffer`

`Kernel.IO.Buffer` is the owned contiguous off-heap buffer.

It should:

- expose readable and writable regions as `IoSlice`
- support growth, compaction, append, and consume
- be the storage primitive that `Std.IO.Buffer` re-exports as the default buffer

## 2. `Std.IO` surface

### 2.1 One public error type

`Std.IO` now exposes one public closed error type and one public result alias:

- `Std.IO.Error`
- `Std.IO.result`

This is the right simplification for `std`.

The previous reader and writer error parameters were technically flexible but
practically noisy. They leaked source details into APIs that should have been
easy to compose. One public `IO.Error` makes these operations straightforward:

- `Reader.chain`
- `Reader.take`
- `BufReader.reset`
- `stdin`, `stdout`, and `stderr` adapters

It also makes `Std.IO` teach one error vocabulary instead of forcing callers to
thread source-specific error types through transport code.

`Kernel.IO.Error` still exists where low-level storage and bounds errors matter.
`Std.IO` converts that kernel-facing detail into the public I/O error model.

### 2.2 `Std.IO.Buffer`

`Std.IO.Buffer` is the default standard-library buffer. It is off-heap and
transport-facing.

That is a deliberate inversion of the old `Std.IO.Buffer` meaning. The old heap
text builder is no longer the default buffer in `Std.IO`; that role now belongs
to the off-heap transport buffer.

### 2.3 `Std.StringBuilder`

`Std.StringBuilder` is the explicit heap text accumulator.

It exists for:

- `Reader.read_to_string`
- formatting and rendering code
- cases where the caller intentionally wants heap-owned text accumulation

It is not the default transport buffer.

### 2.4 `Std.IO.Reader`

`Reader` is buffer-first. Its public surface is:

- `read`
- `read_vectored`
- `is_read_vectored`
- `read_to_end`
- `read_to_string`
- `read_exact`
- `bytes`
- `chain`
- `take`

The critical rule is that raw `Reader.read` does *not* return a borrowed slice.
The caller owns the destination.

### 2.5 `Std.IO.BufReader`

`BufReader` is the one borrowed-slice layer in `std`. Its public surface is:

- `from_reader`
- `to_reader`
- `read`
- `read_byte`
- `size`
- `reset`
- `fill`
- `peek`
- `consume`
- `read_rune`
- `read_slice`
- `read_line`
- `read_string`

Important semantics:

- `peek ~len` is exact-or-error
- `consume ~len` advances by the smaller of the requested count and the current
  readable bytes
- `read_slice ~until` returns a borrowed slice including the delimiter when it
  finds one
- `read_slice ~until` returns `Buffer_full` if the internal buffer fills before
  the delimiter appears
- `read_slice ~until` returns the remaining tail at EOF if any bytes were still
  buffered, otherwise `End_of_file`

### 2.6 `Std.IO.Writer`

`Writer` is the symmetric buffer-first write surface:

- `write`
- `write_vectored`
- `write_all`
- `write_all_vectored`
- `flush`

The core currency is `Buffer` and `IoVec`, not heap strings.

## 3. Ownership and borrowing rules

Borrowed slices are only useful if their lifetime rules are explicit.

This RFD standardizes the following rule:

- a slice borrowed from a caller-owned `Std.IO.Buffer` is valid until that
  buffer is mutated or dropped
- a slice borrowed from `Std.IO.BufReader` is valid until the next operation
  that may refill, consume, or reset the internal buffer

That is the same contract shape as:

- Go `bufio.Reader.ReadSlice`
- Rust `BufRead::fill_buf` plus `consume`

If a caller wants to keep a value longer, it must either:

- keep the owning buffer alive and stable
- or copy with `IoSlice.to_string`, `IoSlice.to_bytes`, or `IoSlice.clone`

## 4. Syscall boundary rules

At the blocking syscall boundary:

- payload pointers retained across the blocking section must come from off-heap
  storage
- heap `string` and heap `bytes` payload pointers must not be retained across
  `caml_enter_blocking_section()`
- vectored operations should use `IoVec` and `IoSlice`, not ad hoc OCaml heap
  slices

That means the correct low-level path is:

1. allocate or reuse off-heap buffers
2. expose `IoSlice` or `IoVec` views
3. call the kernel
4. materialize heap values only after the I/O boundary when ownership truly changes

## 5. HTTP and parser layering

The current HTTP parser results reinforce the intended boundary.

- parser-only `100 KiB` request:
  - baseline string parser: `174.55 us`
  - `parse_slice`: `10.70 us`
- reader-driven full-request `100 KiB` request:
  - baseline string parser: `508.38 us`
  - `parse_slice`: `169.70 us`

But small and header-heavy requests still favor the old string-native baseline:

- reader-driven `small request`: baseline `4.51 us`, `parse_slice` `10.02 us`
- reader-driven GitHub-like request: baseline `97.85 us`, `parse_slice`
  `179.38 us`

So the design rule is:

- transport and buffering should be slice-first
- public HTTP request and response values should remain owned at the boundary
- bodies should stay lazy
- borrowed parser entry points such as `Http1.Request.Borrowed.parse` are additive fast paths,
  not the default public contract

## 6. Migration stance

This RFD does not require a giant borrowed rewrite of Riot.

The intended rollout is:

1. stabilize `Kernel.IO.Buffer`, `Kernel.IO.IoSlice`, and `Kernel.IO.IoVec`
2. make `Std.IO.Buffer` the default `std` buffer
3. keep `Reader` and `Writer` buffer-first
4. keep `BufReader` as the single borrowed-slice layer in `std`
5. migrate parsers onto slices where the benchmark evidence justifies it
6. keep public owned surfaces where they remain easier to use and not obviously slower

## Drawbacks
[drawbacks]: #drawbacks

The main cost of this design is cognitive load.

Contributors now need to reason about:

- the difference between owned buffers and borrowed slices
- explicit ownership boundaries
- borrowed-slice lifetime rules
- checked versus `_unchecked` operations

There is also a real API cost in standardizing on one public `IO.Error`.
Source-specific error detail is no longer exposed directly through
`Std.IO.Reader` and `Std.IO.Writer`. That is the right simplification for `std`,
but it means the public error enum has to be broad enough to cover buffered I/O,
stream state, and OS-adjacent failures.

Finally, the benchmarks are clear that slices are not a universal speed win.
The old string parser still wins on tiny and header-heavy request shapes, which
means Riot must keep measuring instead of assuming that borrowed or off-heap
always means faster.

## Rationale and alternatives
[rationale-and-alternatives]: #rationale-and-alternatives

### Why keep one public `IO.Error`?

Because the previous source and error parameters made the transport APIs harder
to teach and harder to compose than they needed to be.

The important abstraction boundary in `Std.IO` is no longer "what concrete
reader produced this error?" It is "did this standard-library I/O operation
succeed?" A single public error type makes the normal `Std.IO` surface much
cleaner without preventing lower layers from keeping richer internal error
details.

### Why not keep `StringView`?

`StringView` did not justify its own runtime concept once `IoSlice` gained the
helpers we actually needed.

- it duplicated operations that belonged naturally on `IoSlice`
- it added a third concept where two were enough
- it did not solve the real lifetime problem, which is about who owns the
  backing bytes

The final model keeps the text-ish helpers on `IoSlice` and uses explicit
materialization as the ownership boundary.

### Why not make raw `Reader.read` return a slice?

Because that makes ownership less clear, not more.

If raw `read` returns a slice, then either:

- the reader allocates a fresh buffer every call
- or it returns a borrowed slice from hidden internal storage

The first hides allocation in the hot path. The second hides borrowing in the
raw reader contract. The cleaner split is:

- `Reader`: caller-buffered
- `BufReader`: borrowed-slice layer

### Why not keep the old string-first standard-library surface?

Because it taught the wrong default.

String-first helpers remain useful as explicit adapters. They should not be the
foundation of `Std.IO`. Making off-heap buffers the default changes what Riot
contributors reach for first, which is exactly the leverage this redesign
needs.

### Why not make every public parser borrowed?

Because the benchmark evidence does not justify that complexity for every API.

The right outcome is:

- slice-first transport and buffering
- borrowed parser entry points where they measurably help
- owned public protocol values where that remains easier to use and not
  clearly slower

### What happens if Riot does nothing?

Riot keeps paying the same recurring costs:

- hidden transport copies
- mixed buffer ownership stories
- awkward `Std.IO` composition points
- more drift back toward heap strings in places that should stay off-heap

## Prior art
[prior-art]: #prior-art

Riot is converging on the same split used by other high-performance systems,
while still choosing its own public ergonomics.

### Go

- `io.Reader.Read(p []byte)` fills caller-owned buffers
- `bufio.Reader.ReadSlice(delim)` returns borrowed slices from an internal buffer
- converting `[]byte` to `string` is an explicit ownership change

### Rust

- `Read::read(&mut [u8])`
- `Read::read_vectored(...)`
- `Read::read_to_end(...)`
- `BufRead::fill_buf()`
- `BufRead::consume(n)`

Rust enforces more of the borrowing story in types. Riot documents the same
rules at the API boundary.

### OCaml ecosystem

- `bigstringaf` provides stable off-heap storage
- Eio's bigstring and `Cstruct` flow model keeps transport bytes off the OCaml heap
- higher-level HTTP stacks commonly keep bodies lazy or streaming instead of
  materializing them during head parsing

## Unresolved questions
[unresolved-questions]: #unresolved-questions

- Should Riot add a `BufWriter` symmetric to `BufReader`?
- Should `IO.Types.u8` remain a simple alias for now, or become a more explicit
  byte-level public type later?
- Should Riot add a generic incremental parser substrate later, or keep parser
  state machines package-specific?
- Should long-lived borrowed values eventually gain a first-class retained-buffer
  abstraction instead of relying on "keep the owning buffer alive"?

## Future possibilities
[future-possibilities]: #future-possibilities

Future follow-up work may include:

- `BufWriter` and other buffered write helpers
- incremental HTTP head parsers and other stateful streaming parsers on top of
  `BufReader`
- retained-view or chunk-retention abstractions for long-lived borrowed values
- more parser migrations where benchmarks justify the extra complexity
- clearer public docs that compare Riot's model directly to Go and Rust
