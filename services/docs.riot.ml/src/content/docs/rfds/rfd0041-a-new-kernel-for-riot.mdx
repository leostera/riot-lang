## Summary
[summary]: #summary

This RFD proposes a new kernel package for the Riot stack to build on. This
kernel will be Riot's platform abstraction layer, with only the minimum
foundational types and primitives needed to define that layer and to let `std`
build on it without caring about the operating systems or architectures being
targeted. It will be self-contained, not depend on Stdlib/Unix and other stock
OCaml libraries, and expose APIs shaped so `std` can wrap them almost
verbatim.

## Motivation
[motivation]: #motivation

The primary trigger for creating a new kernel _soon_ is our work on the `typ`
incremental type checker. Checking the existing kernel, which depends on
Stdlib/Unix/Dynlink/etc from stock OCaml, means we suddenly need to support a
very large language surface. A simpler kernel, that is self-contained and
doesn't leak platform stock OCaml abstractions, would help with this.

On its own, the current kernel has several deficiencies:

- as a layer on top of Stdlib and Unix, it often leaks abstractions that aren't
  Riot-friendly, such as exceptions or blocking operations
- it is hard to optimize, since most of the primitives live in OCaml itself
- in some cases they force an indirection cost when we want to put
  friendlier, more modern APIs on top of legacy ones
- Unix is also Unix-shaped, which makes cross-platform kernel work awkward when
  Riot wants to expose one contract over multiple platform-specific syscalls,
  such as supporting operations like `clonefile` on macOS without letting the
  public kernel surface become a patchwork of OS-specific legacy concepts

The goal is not to make `kernel` a second full standard library. The goal is to
make `kernel` the narrow, Riot-owned platform abstraction layer that `std` can
build on. `std` should contain the normal library surface Riot users program
against. `kernel` should contain the things `std` should not have to
re-implement per operating system, architecture, or runtime backend.

## Guide-level explanation
[guide-level-explanation]: #guide-level-explanation

Suppose a Riot contributor wants to add vectored I/O support for files.

Today, that work starts in the wrong place. Instead of beginning from a Riot
file or socket contract, the contributor has to drop into C, add
`readv`/`writev` bindings, thread `Unix.file_descr` through the kernel
boundary, make sure the new calls fit with the rest of the Unix-shaped file and
socket code, map Unix errors back into Riot errors, and only then wrap the
result again so `std` can expose it ergonomically.

```ocaml
(* kernel code today *)
module Fd : sig
  type t
  val to_unix : t -> Unix.file_descr
  val of_unix : Unix.file_descr -> t
end

module Fs.File : sig
  type t = Fd.t
  val read_vectored : t -> IO.Iovec.t -> (int, IO.error) result
  val write_vectored : t -> IO.Iovec.t -> (int, IO.error) result
end

external std_sys_readv : Unix.file_descr -> IO.Iovec.t -> int = "kernel_unix_readv"
external std_sys_writev : Unix.file_descr -> IO.Iovec.t -> int = "kernel_unix_writev"

(* std code today *)
module IO.Reader : sig
  val read_vectored : ('src, 'err) t -> Iovec.t -> (int, 'err) result
end

module IO.Writer : sig
  val write_owned_vectored : ('dst, 'err) t -> bufs:Iovec.t -> (int, 'err) result
end
```

The painful part is not the syscall itself. The painful part is that Riot has
to extend the kernel through Unix-shaped seams first, and only afterwards shape
the result into the API Riot actually wanted.

The same discomfort shows up in cross-platform work. If Riot wants one file API
that can use `clonefile` on macOS, a different copy primitive on Linux, and a
third implementation later in Zig, the current kernel makes that awkward
because the public API keeps starting from platform concepts instead of from a
Riot-owned contract.

With the new kernel, the same extension should begin by defining Riot's own
kernel file contract first, and let `std` build reader and writer protocols on
top of it:

```ocaml
open Kernel

module Fs.File : sig
  type t
  val read_vectored : t -> IO.Iovec.t -> (int, error) result
  val write_vectored : t -> IO.Iovec.t -> (int, error) result
end

(* std *)
module IO.Reader : sig
  val read_vectored : ('src, 'err) t -> Iovec.t -> (int, 'err) result
end

module IO.Writer : sig
  val write_owned_vectored : ('dst, 'err) t -> bufs:Iovec.t -> (int, 'err) result
end
```

The important part is not the exact function names. It is the design center:

- the contributor starts from Riot's own kernel file and socket contract
- files, sockets, and future backends all implement that same contract
- platform-specific syscalls stay behind the backend instead of shaping the
  public API
- `std` can expose reader and writer protocols on top, with actor-friendly
  suspension layered on top where needed

## Reference-level explanation
[reference-level-explanation]: #reference-level-explanation

## 1. Package contract

This new kernel is a new package, not a migration mode for the current kernel.

The primary design goal is ownership: Riot should own the kernel's public API,
its internal abstractions, and the direction of its native implementation.

More specifically, the new kernel should be Riot's platform abstraction layer.
It should provide the minimum foundational types needed to express that layer,
and the platform-dependent primitives that `std` should be able to build on
without having to care about Unix, Windows, Wasm, JavaScript, or future
backends directly.

It should obey these package-level rules:

- no Riot package dependencies on `stdlib`, `unix`, or `dynlink`
- no public signatures mentioning `Stdlib.*` or `Unix.*`
- Riot-owned implementation and public contract end to end
- no public `panic`, `unwrap`, or implicit-exception contract for fallible
  system operations

The only runtime support it may rely on is:

- compiler/runtime intrinsics exposed directly through externals or primitives
- native stubs Riot owns
- toolchain-shipped compiler support modules that are not treated as Riot
  package dependencies

## 2. Scope

The new kernel should be intentionally smaller than the current kernel, and it
should be much narrower than `std`.

### In scope for the initial package

- primitive scalar modules Riot needs as stable homes for compiler-provided
  types at the kernel boundary:
  `Bool`, `Char`, `Int`, `Int32`, `Int64`, `Float`
- minimal foundational types needed to express kernel APIs:
  `String`, `Bytes`, `Array`, `Option`, `Result`, `Error`
- path and byte-vector primitives needed by kernel APIs:
  `Path`, `IO.Iovec`
- filesystem primitives, especially file handles
- network address types and socket handles
- raw system time primitives
- async registration / readiness primitives
- environment and process primitives
- effect/runtime hooks needed to let `std` and the rest of Riot build on the
  kernel cleanly

### Explicitly out of scope

- a full stdlib replacement
- full cloned collection modules like `List`, `Array`, `Map`, `Seq`, `Random`
- iterator protocols and higher-level library patterns
- `Reader` and `Writer` as general user-facing protocols
- compatibility aliases like `to_unix`, `of_unix`, `*_to_unix`
- top-level `Global` flattening as a public convenience surface
- re-exporting `Dynlink`, `Printexc`, `Marshal`, or other OCaml runtime modules
  just because they exist
- regex, compression, archive, and crypto libraries unless they later prove to
  be unavoidable kernel concerns

If Riot later wants owned collections or richer general-purpose modules, they
should live in `std` or a separate foundational package, not be smuggled into
the new kernel by inertia. The constraint is about purpose, not ideology: the
new kernel should not become a general-purpose standard library clone.

## 3. Internal layering

The new kernel should have an explicit internal split.

### 3.1 Bootstrap layer

This layer is internal-only and minimal.

It exists to bind:

- compiler intrinsics
- primitive array/string/bytes operations
- low-level comparison and arithmetic intrinsics
- runtime support needed to let the rest of the new kernel compile without pulling
  in `Stdlib`

This layer should not be the public API. It is the equivalent of Riot's
bootloader for the package.

### 3.2 Core owned modules

This layer defines Riot-owned types and helper modules. These modules should be
designed around what Riot actually needs, not around preserving stdlib
documentation or surface area.

Examples:

- `Result` should be explicit and compositional, but it does not need every
  stdlib-inspired convenience that panics
- `Option` should follow the same rule
- string/bytes/array APIs should be sized to kernel needs and kernel-facing
  contracts, not to parity with all stdlib helpers
- `Path` and `IO.Iovec` belong here because kernel modules need them to express
  portable APIs
- this layer exists to support the kernel boundary, not to grow general-purpose
  library abstractions

### 3.3 Systems layer

This is the real kernel surface:

- `IO.Iovec`
- `Fs.File`
- `Net`
- `Time`
- `Env`
- `Process`
- `Async`
- `Effect`

These modules should be designed so `std` can mostly wrap or re-export them,
not reinterpret them.

## 4. Public API rules

Every public module in the new kernel should satisfy these rules.

### 4.1 Owned types only

Public signatures must not expose:

- `Unix.file_descr`
- `Unix.stats`
- `Unix.error`
- `Unix.socket_domain`
- `Unix.socket_type`
- `Stdlib.Mutex.t`
- `Stdlib.Sys.signal_behavior`
- similar OCaml-runtime implementation types

Instead, the new kernel should define:

- abstract handles for files, sockets, listeners, timers, and processes
- owned metadata records
- owned address types
- owned signal and process behavior enums where needed

### 4.2 Explicit fallibility

Public operations that can fail should return results with Riot-owned errors.

That means:

- no public `Unix.Unix_error`
- no native `caml_failwith` contracts
- no public APIs that silently swallow write or flush failures
- no panic-driven extraction helpers in the core systems APIs

Programmer-bug assertions may still exist internally, but the kernel contract
for system operations should be result-based.

### 4.3 No public compatibility conversions

The new kernel should not expose public compatibility conversions like:

- `Fd.to_unix`
- `Fd.of_unix`
- `seek_command_to_unix`
- `file_kind_of_unix`
- similar leak-through adapters

If the implementation needs them internally, they stay internal to the backend.

### 4.4 Stable native boundary

The native boundary should be designed as if Zig were already a future backend.

That means:

- explicit widths and layouts
- opaque integer or pointer handles wrapped by OCaml abstract types
- borrowed buffers and owned buffers called out explicitly
- no dependence on OCaml exceptions for routine error reporting

## 5. Native implementation strategy

The initial backend may still be written in C, but it should follow Zig-ready
rules.

Native functions should prefer contracts like:

- return status code
- write outputs into passed buffers or out-parameters
- map OS errors into explicit kernel error codes

over contracts like:

- call libc
- raise `caml_failwith` on error
- let OCaml decode the exception path later

This matters because the hard part of a Zig rewrite is not syntax conversion.
It is whether the boundary is already clean.

## 6. Relationship to `std`

`std` remains Riot's normal application-facing surface.

The new kernel should not try to be `std`. It should be the platform
abstraction layer that lets `std` exist without having to think about operating
systems, architecture-specific runtime details, or native handles directly.

Where the same concept exists in both layers, the naming and shape should stay
as close as practical. `std` should be able to say:

- "this is the same file handle idea, but with more application-facing policy"
- "this is the same socket idea, but with more ergonomic helpers and actor
  integration"
- "this is the same operation, but wrapped in the standard Riot concurrency
  model"

It should not have to say:

- "this is the Unix-shaped kernel thing, translated into Riot names"

## Drawbacks
[drawbacks]: #drawbacks

- this is a large reset and cannot be implemented as a small cleanup branch
- Riot will have to temporarily maintain an old kernel and a new kernel in
  parallel while consumers move
- owning the entire kernel means Riot must own more boring foundational work
  itself
- a smaller kernel means some modules people expect to "obviously live in
  kernel" will need to move or wait
- the bootstrap layer will still be subtle and easy to get wrong if it grows
  beyond its intended scope

## Rationale and alternatives
[rationale-and-alternatives]: #rationale-and-alternatives

### Do nothing

Riot could keep `kernel` as the low-level boundary and continue the current
incremental cleanup approach.

This was rejected because the audit already shows that the current cleanup
approach has inherited too much of the old structure:

- package deps still include `stdlib`, `unix`, and `dynlink`
- public APIs still leak Unix
- inherited implementation still defines much of the package
- native code still mixes Riot-owned and legacy assumptions

That makes "incremental cleanup" likely to preserve the wrong design center.

### Keep rewriting the current kernel

This was rejected because it would still optimize for continuity with a package
whose current shape is exactly what Riot wants to leave behind.

The current cleanup work is useful as evidence:

- what low-level capabilities Riot needs
- which APIs feel good
- which areas are still Unix- and stdlib-shaped

But it is not a good implementation base for the new kernel.

### Build a compatibility layer first

This was rejected explicitly.

The goal of the new kernel is not to provide a migration story through inherited
surface area. The goal is to define the correct kernel contract. A compatibility
layer would lock in old shapes precisely where Riot wants freedom to redesign.

### Start with Zig first, design later

This was rejected because the language of the backend is not the main decision.
If the public contract is wrong, a Zig port will just faithfully preserve the
wrong contract.

The right order is:

1. define the owned kernel contract
2. implement it with a clean native boundary
3. rewrite the backend later if Riot wants Zig

## Prior art
[prior-art]: #prior-art

- Riot's current `kernel` shows the existing low-level package boundary but is
  intentionally allowed to depend on `stdlib` and `unix`
- the current cleanup work shows both the opportunity and the failure mode: it
  improves some ergonomics but still inherits stdlib, Unix, and too much
  legacy structure
- Rust's internal `sys` layering is useful as a reminder that one public file
  or socket API can still hide very different platform implementations
- Zig's standard library is relevant less as an API template and more as a
  reminder that low-level libraries can be designed around explicit error and
  ownership rules

## Unresolved questions
[unresolved-questions]: #unresolved-questions

- What is the exact minimal bootstrap surface the new kernel needs in order to
  compile without `stdlib`?
- Which current low-level modules are truly kernel concerns, and which should
  move to `std` or a separate base package?
- Which socket operations should live directly on kernel socket handles, and
  which should be introduced higher in `std`?
- How much of time should stay in the kernel beyond raw system time?
- What is Riot's exact policy for panic-style helpers in foundational pure
  modules like `Option` and `Result`?
- How should the toolchain ship compiler/runtime support modules for
  the new kernel's bootstrap layer?

## Future possibilities
[future-possibilities]: #future-possibilities

- a Zig backend for selected or all native subsystems without public API
  breakage
- a Windows-native backend once the contract is no longer Unix-shaped
- a separate Riot-owned foundational package for collections and richer pure
  data structures
- richer `std`-level protocols built on the kernel substrate, such as reader
  and writer traits with actor-friendly integration
- code generation for repetitive native binding layers once the new kernel
  boundary stabilizes
- a stricter `no-stdlib` toolchain mode that relies on the new kernel as the real
  substrate instead of on inherited OCaml library behavior
