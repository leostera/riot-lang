## Summary
[summary]: #summary

This RFD proposes changing `riot-planner` from a package-centric module
selection model to a target-centric one. Riot should still scan one package
module graph, but instead of deciding inclusion only at the library boundary,
it should compute a reachable closure per target root: the library root, each
binary root, and eventually each test, bench, example, or command root. This
is a planner and build-graph change, not a new package format and not a new
visibility system in the language.

- A package library should archive only the modules reachable from its concrete
  library root.
- Each runnable target should get its own reachable closure and be allowed to
  link target-private helper modules without forcing them into the package
  library.
- Unreached modules should not be compiled or linked at all.
- Recursive source scanning stays in place; this RFD changes ownership and
  action planning, not how Riot discovers files.
- New package syntax, explicit visibility annotations, and target-specific
  compile flags are out of scope.

## Motivation
[motivation]: #motivation

Riot's current planner is still mostly package-centric.

The recent library reachability work improved one real problem: files that are
not reachable from the library root no longer need to be archived into the
package library by default. That makes the library artifact better reflect
actual library structure.

But the rest of the planner still behaves like this:

- the package has one main reusable artifact, its library
- binaries are thin roots that compile their own entry file and then link
  against that library
- helper modules that are not part of the library have nowhere principled to go

That shape keeps showing up as practical contributor friction.

### Riot cannot express target-private helper modules cleanly

Suppose a package looks like this:

```text
src/lib.ml
src/a.ml
src/b.ml
src/main.ml
```

and:

- `lib.ml` reaches `A`
- `main.ml` reaches `B`

What contributors want is obvious:

- `A` should be part of the package library
- `B` should be built for the binary
- `B` should not be forced into the exported library

Today that is not the planner model. Once Riot trims `B` out of the library, a
binary root still does not have its own closure to pull `B` back in. The
contributor either has to:

- move `B` into the library even though it is not library API,
- create a separate package just to hold binary helpers, or
- avoid splitting the entrypoint into smaller modules in the first place

That is the wrong tradeoff. Riot should make it easy to split work into helper
modules without treating every helper as library surface area.

### Tests, benches, examples, and commands want the same thing

This is not only a `src/main.ml` problem.

The same structural need exists for:

- test-suite helpers
- benchmark helpers
- example-local helpers
- command-local helpers

RFD0035 already moves Riot toward a more natural multi-file target layout under
`tests/`, `bench/`, and `examples/`. That layout becomes much more useful if
the planner can say:

- these modules belong to the test root
- these modules belong to the bench root
- these modules belong to the example root
- these modules are dead and should not be built

Without that, Riot still pays for a recursive source tree while only one target
class, the package library, has a real notion of ownership.

### Riot currently makes in-progress files more expensive than they need to be

A contributor often creates new modules before wiring them into the final root:

```text
src/new_parser.ml
src/new_parser_state.ml
```

Those files should be cheap to stage while the contributor is working on them.

Today the planner pressure is still too high:

- if Riot includes them too eagerly, they can fail the build before they are
  actually used
- if Riot trims them at the library boundary, there is still no parallel story
  for binary/test/example roots

The real cost Riot pays is that source-tree structure and build ownership do
not line up cleanly. Contributors keep paying with extra package ceremony,
library pollution, or reluctance to split code into smaller units.

### Doing nothing keeps the planner at the wrong level

If Riot does nothing, this cost will recur every time we want any of these:

- multi-file binaries with local helpers
- multi-file tests, benches, and examples
- smaller entrypoint files
- dead-module trimming that reflects actual target usage instead of just
  library usage

Riot will keep having one good reachability rule for libraries and a weaker,
special-case rule for everything else. That is the wrong long-term shape for
`riot-planner`.

### This is also the right base for conditional compilation

Riot does not have full conditional compilation yet, but when it does, the
planner will no longer be able to treat the package graph as one fixed,
package-wide fact.

After conditional erasure, the effective module graph may differ by:

- target
- profile
- workspace or package configuration
- active root

That matters here because reachability should be computed from the resolved
graph that Riot will actually build, not from the raw file tree before target-
specific resolution.

Target-specific reachability gives Riot the correct place to stand when that
future arrives:

- resolve the graph for this build context
- compute closures from that resolved graph
- build only what those roots actually reach

That is a better long-term planner model than continuing to treat the package
library as the only root with real ownership semantics.

## Guide-level explanation
[guide-level-explanation]: #guide-level-explanation

With this proposal, contributors should think of a package as having:

1. one shared module graph
2. several target roots over that graph
3. one reachable closure per target root

The target roots are things Riot can build directly:

- the library root
- each binary root
- each test root
- each benchmark root
- each example root
- each command root

### Example: library plus binary-private helper

Suppose a package has:

```text
src/lib.ml
src/a.ml
src/b.ml
src/main.ml
src/orphan.ml
```

and:

- `lib.ml` uses `A`
- `main.ml` uses `B`
- nothing uses `orphan.ml`

With this proposal, Riot should plan that package like this:

- `library_closure = { lib.ml, a.ml }`
- `binary_closure(main) = { main.ml, b.ml }`
- `dead = { orphan.ml }`

Then Riot should:

- archive `A` into the package library
- compile and link `B` into the executable for `main`
- not build `orphan.ml` at all

The contributor mental model becomes simple:

- files reachable from the library root are library code
- files reachable only from a runnable target are target-private helpers
- files reachable from nothing are dead

### Example: test-local helpers

Suppose a package has:

```text
src/lib.ml
src/a.ml
tests/parser/main.ml
tests/parser/helpers.ml
tests/parser/fixtures.ml
```

With this proposal:

- `helpers.ml` and `fixtures.ml` should be allowed to exist purely for the
  `parser` test target
- they should not be forced into the package library
- they should not become separate test suites

That gives Riot a coherent story for multi-file targets:

- recursive scan still discovers the files
- target roots decide who owns them
- the action graph only compiles what a real target reaches

### Important consequence

This feature is not a visibility system in source code. Contributors do not
annotate modules as "private to main" or "private to tests".

Instead, Riot derives that classification from ordinary dependency reachability
over the package graph.

That keeps the model aligned with the recent library reachability work:

- use real `Syn.Deps` edges
- do not guess from directory shape
- do not add CST or path-based fallback ownership rules

## Reference-level explanation
[reference-level-explanation]: #reference-level-explanation

## 1. Planner model

`riot-planner` should move from:

- one package graph
- one special reachable closure for the library
- thin executable roots that just link the library

to:

- one package module graph
- one target root record per buildable target
- one reachable closure per target root

This is still a package-local planning pass. It does not require package
splitting and does not change dependency solving.

The important semantic rule is that reachability is computed from the resolved
module graph for the current build context, not from the raw discovered source
forest. If Riot later erases modules or edges through conditional compilation,
those changes must happen before closure computation.

## 2. Target roots

The planner should treat the following as target roots:

- the concrete library root when a package has a library
- each package binary root
- each command root
- each autodiscovered test root
- each autodiscovered bench root
- each autodiscovered example root

The exact target inventory already differs by scope. This RFD does not change
scope semantics; it changes how module ownership is derived once a target root
exists.

## 3. Reachability sets

For each target root `T`, Riot computes:

- `reachable(T)`: all modules reachable from `T` using resolved `Syn.Deps`
  edges plus the structural support edges already required for correct
  compilation, such as `ML -> MLI`

For the package library root `L`, Riot computes:

- `library_reachable = reachable(L)`

For each non-library target root `T`, Riot computes:

- `target_reachable(T) = reachable(T)`
- `target_private(T) = target_reachable(T) - library_reachable`

Modules not in any reachable set are dead:

- `dead = all_modules - union(all reachable sets)`

## 4. Action ownership

This proposal does not require a separate scan or separate source bucket per
target. The important change is action ownership.

### Library

`CreateLibrary` should consume only the objects for `library_reachable`.

### Executable-like targets

For each binary, command, test suite, benchmark suite, or example root, the
final link action should consume:

- the root object for that target
- the objects in `target_private(T)`
- the package library archive when the package has a library
- transitive external libraries as today

This gives the executable its own private helper closure without forcing those
helpers into the archive.

## 5. Binary roots stop being thin link stubs

Today a binary node is effectively a link target with one compiled source file
and a dependency on the package library.

With this proposal, binary-like targets must become real reachability roots in
the planner model. That means Riot must be able to:

- analyze the target root source as a normal module
- compute its resolved dependency closure
- retain private helper modules for that target

The package graph stays shared; the ownership and final link inputs change.

## 6. Interaction with recent library reachability work

The recent library trimming work should remain the base invariant:

- library closure is derived from the concrete library root
- resolved `Syn.Deps` edges are authoritative
- there is no CST fallback and no directory-shape fallback

This RFD generalizes that rule instead of replacing it.

Where Riot currently asks:

- what is reachable from the library root?

it should start asking:

- what is reachable from each target root?

## 7. Interaction with RFD0035

RFD0035 proposes a cleaner target layout for tests, benches, and examples.

That proposal becomes much more valuable with this one, because nested files
under a target directory stop being ambiguous:

- they are not separate runnable targets
- they are not accidental library files
- they are ordinary support modules owned by that target if they are reachable

These RFDs are independent but complementary:

- RFD0035 changes target discovery rules
- this RFD changes target ownership and action planning

## 8. Implementation boundary

The implementation should live in `riot-planner`.

`riot-build` should not invent separate target-private heuristics at execution
time. By the time execution starts, the action graph should already say:

- which modules belong to the library
- which modules belong to each runnable target
- which modules are dead

## Drawbacks
[drawbacks]: #drawbacks

- The planner model becomes more complex because it has to reason about more
  than one reachable closure per package.
- Action ownership becomes harder to debug. A missing module may now be absent
  because it is dead, because it only belongs to another target, or because
  dependency extraction is wrong.
- This raises the quality bar for `Syn.Deps`. If dependency extraction misses a
  real edge, Riot may incorrectly trim a needed helper module.
- Link planning becomes more subtle because executable-like targets now own
  private object sets in addition to libraries.

## Rationale and alternatives
[rationale-and-alternatives]: #rationale-and-alternatives

### Do nothing

Riot could keep the current model:

- library reachability trimming for libraries only
- thin executable roots for binaries/tests/benches/examples

That is simpler in the short term, but it leaves the main contributor problem
in place. Helpers still have no principled home unless they are promoted into
the library or split into separate packages.

### Keep everything package-centric

Riot could continue compiling most modules at package scope and only trim a few
obvious dead cases.

That avoids planner complexity, but it loses the main benefit:

- helper modules still pollute the package build
- dead files remain too expensive
- multi-file targets still do not have a clean ownership model

### Require separate packages for helper code

Riot could say that binary-local or test-local helpers should live in separate
packages.

That is too much ceremony for the common case. It turns ordinary internal code
organization into package management and makes small refactors disproportionately
heavy.

### Introduce explicit visibility annotations

Riot could add package metadata or source annotations to say that a module is
"private to target X".

That is a larger, more user-facing system than the problem requires. The
planner already has a dependency graph. If ordinary reachability can express
ownership, Riot should prefer that over introducing a second configuration
language.

### Why this design

This design is the best fit because it generalizes a rule Riot already wants:

- use the real dependency graph
- build only what real roots reach

It removes a structural asymmetry instead of adding more target-specific
exceptions.

## Prior art
[prior-art]: #prior-art

### Dune

Dune models libraries and executables as separate stanzas with separate module
sets. That is not the same implementation model Riot uses, but it does show
the value of treating library code and executable-private code as different
ownership classes.

### Cargo

Cargo's test, bench, and example layouts distinguish runnable roots from
support modules under those trees. That is a useful precedent for Riot's
multi-file target story, especially alongside RFD0035.

### Riot today

Riot already has the beginnings of this model:

- recursive source scanning
- resolved syntactic dependency extraction via `Syn.Deps`
- library reachability trimming from a concrete library root

This RFD proposes finishing that generalization instead of leaving it
library-only.

## Unresolved questions
[unresolved-questions]: #unresolved-questions

- Should target-private closures be materialized as explicit planner-owned
  target records, or derived on demand during action-graph construction?
- Should compile actions for modules reused by multiple targets be shared
  structurally whenever their inputs are identical, or should the first version
  optimize only ownership and leave compile sharing unchanged?
- How should Riot render target-private ownership in debug output so planning
  mistakes are diagnosable?
- Should dead modules be completely absent from the action graph, or should Riot
  retain a planner-visible classification for diagnostics and future tooling?

## Future possibilities
[future-possibilities]: #future-possibilities

This target-centric reachability model would be a good base for:

- better support for multi-file binaries under `src/`
- more natural support-module layouts for tests, benches, and examples
- conditional compilation that changes the effective module graph after target-
  or profile-specific resolution
- planner diagnostics that explain why a file is dead or who owns it
- future editor or `riot info` tooling that can show per-target module
  membership
- more precise incremental planning and invalidation because Riot can track
  which targets actually care about a changed module
