## Summary
[summary]: #summary

This RFD proposes rewriting `riot-build` and the workspace executor around a
typed, in-process build session with one unified work scheduler for package
planning and action execution. The new design removes the leftover
request/response local-session architecture, makes build ordering explicit in
types, and lets Riot start useful work as soon as any package or action becomes
ready instead of forcing the whole build through separate orchestration phases.

- `riot-build` consumes a prepared `Workspace.t`; workspace discovery,
  dependency resolution, external source loading, and package creation stay
  outside the build package.
- Build execution is modeled as typed stages: request resolution, lane locking,
  target-aware lane preparation, package planning, action execution, package
  finalization, and cache generation recording.
- A single scheduler owns both package-planning work and action-execution work
  under one concurrency budget, across all requested targets.
- Every work item carries its target/profile compilation lane so future
  conditional compilation can make planning target-aware.
- This RFD does not define conditional-compilation syntax, remote execution,
  remote cache behavior, or new CLI flags.

## Motivation
[motivation]: #motivation

Riot's current build path is one-shot and local, but part of its architecture is
still shaped like a small client/server system. That made sense when the build
system had a session protocol boundary. It is now mostly baggage.

Today `riot-build` still has a request/response-shaped internal contract, a
session loop, and a spawned build worker. Even after module names are cleaned
up, the underlying shape remains:

- a caller sends a build request into an internal mailbox
- a runtime loop receives it
- a worker process emits responses back through that mailbox
- `build_runtime` waits for a final response and only then returns results

That is structural complexity in a package that should now be a direct local
build library. The caller already has a prepared workspace and a typed build
request. There is no external peer, no durable daemon, and no transport boundary
to protect.

The current design also makes it too easy to hide work behind a quiet boundary.
One local observation while debugging this path was:

```text
target_build_started at 414us
target_build_finished at 4,027,217us
```

for `./riot-old run riot -- build riot-cli --json`. During that gap, the build
command had no build-domain phase events explaining whether Riot was planning,
checking cache, activating packages, executing actions, or finalizing results.
Some executor telemetry exists, but the build-facing event bridge can drop it
instead of making the build command's structured output explain where time is
going. That is a design smell: local build orchestration should not need a
message bridge to learn what it is doing.

The executor already contains part of the architecture Riot wants. In
`packages/riot-executor/src/coordinator.ml`, `build_workspace_actions` has:

- a package-planning state machine
- a pending-planning map
- per-package action queues
- one workspace-level ready-action queue
- one worker count derived from `Build_ctx.parallelism`

That is close to the right direction, but it is not yet unified. Package
planning still feeds action execution from the side instead of being work in
the same scheduler. The build runtime still loops over targets one lane at a
time. A multi-target request such as "build all linux targets" should not
serialize one target lane after another when independent packages and actions
could use the same worker budget.

This becomes more important as Riot moves toward target-aware planning and
conditional compilation. Today it is tempting to think "plan once, execute many
targets", but that model will not survive target-dependent source selection,
dependencies, foreign configuration, or compiler flags. A package planned for
`aarch64-apple-darwin` and the same package planned for `x86_64-unknown-linux`
must be allowed to produce different module graphs and action graphs. The build
scheduler should treat those as separate lane-specific work items from the
start.

The costs Riot pays today are:

- **Extra code and concepts:** local request/response machinery exists where a
  direct function call should be enough.
- **Weak invariants:** the current protocol can represent operations in orders
  that the real build flow should never allow.
- **Late feedback:** planning and scheduler bookkeeping can happen before any
  visible action starts, which delays compiler errors and makes `--json`
  output less useful.
- **Underused parallelism:** package planning, action execution, and target
  lanes are coordinated by separate loops instead of one budgeted scheduler.
- **Future feature pressure:** conditional compilation needs target-aware
  planning, but the current mental model still encourages target-independent
  planning.

If Riot does nothing, each new build feature will have to decide whether it
belongs in the request protocol, the session runtime, the worker, the planner,
the executor, or the CLI. That boundary tax will keep recurring. Multi-target
builds will also remain harder to make fast because the build runtime owns a
target loop above the executor instead of scheduling lane-specific work through
one shared system.

## Guide-level explanation
[guide-level-explanation]: #guide-level-explanation

Suppose a contributor runs:

```sh
riot build riot-cli --target linux
```

The CLI should still own user-facing parsing. By the time `riot-build` is
called, the command has already:

1. found or synthesized the workspace manifest
2. prepared it through `riot-deps.ensure_workspace`
3. parsed package names and target strings at the command edge
4. built a typed `Riot_build.Request.t`

`riot-build` should then behave like a direct local library:

```ocaml
Riot_build.build request ~on_event
```

There is no local protocol and no hidden request loop. The build function
turns the request into executable work.

### The contributor mental model

With this proposal, contributors should think of a build as:

1. resolve requested packages and targets
2. acquire the build lock for each lane that will run
3. create target-specific build lanes
4. seed one scheduler with package-planning work for those lanes
5. let workers plan packages and execute actions under one budget
6. finalize packages as their actions complete
7. record successful cache generations
8. return a typed `Build_result.t`

The most important change is that packages and actions are both work items.
Planning a package is work. Executing an action is work. Finalizing a package is
work. They all flow through one scheduler instead of being split across an
orchestrator loop, a package-planning loop, and a separate action queue.

### Example: one package, one target

For a simple host build, the queue starts with one package-planning item:

```text
PlanPackage { lane = host/debug; package = riot-cli }
```

A worker picks it up.

If the package is fully cached, the worker records a package result and emits a
cached package event. No actions are queued.

If the package needs work, the worker asks `riot-planner` for the target-aware
module graph and action graph. As soon as ready actions exist, it enqueues them
into the same scheduler:

```text
ExecuteAction { lane = host/debug; package = riot-cli; action = compile cli.ml }
ExecuteAction { lane = host/debug; package = riot-cli; action = compile build.ml }
```

Other workers can start those actions immediately. Riot does not need to wait
for every package in the workspace to finish planning before compiling the
first ready module.

### Example: multiple packages

Assume package `app` depends on packages `http` and `jsonrpc`.

The scheduler may start with:

```text
PlanPackage { lane = host/debug; package = http }
PlanPackage { lane = host/debug; package = jsonrpc }
PlanPackage { lane = host/debug; package = app }
```

If `app` is not ready because its dependencies are not built, the scheduler
does not fail. It records that `app` is waiting and lets workers continue on
`http` and `jsonrpc`. When those packages complete successfully, `app` becomes
ready and re-enters the queue.

The key rule is that workers do not own package order. The scheduler owns
readiness. Workers only take ready work.

### Example: multiple targets

For a target pattern such as `--target linux`, Riot may resolve more than one
target triple. Instead of doing one target build at a time, Riot creates one
lane per target:

```text
Lane { profile = debug; target = x86_64-unknown-linux-gnu }
Lane { profile = debug; target = aarch64-unknown-linux-gnu }
```

Then it queues package-planning work per lane:

```text
PlanPackage { lane = x86_64-linux/debug; package = app }
PlanPackage { lane = aarch64-linux/debug; package = app }
```

Each work item carries the lane because planning can eventually depend on the
target. That is what makes conditional compilation possible later without
rewriting the build scheduler again.

### Typed stages

A contributor reading the code should see the build flow in the types:

```ocaml
Request.t
  -> Resolved_build.t
  -> Locked_lane.t list
  -> Scheduler.run
  -> Build_result.t
```

The type names matter. A function that needs a lock should require
`Locked_lane.t`, not a raw target triple. A function that executes an action
should require a planned action work item, not a package manifest. A function
that records cache generations should require completed lane results.

That is the difference between "the code happens to call things in the right
order" and "the compiler makes the wrong order hard to express."

## Reference-level explanation
[reference-level-explanation]: #reference-level-explanation

### Package boundaries

The package responsibilities should be:

- `riot-cli` parses user input and renders events.
- `riot-deps` prepares workspaces and materializes external dependencies.
- `riot-build` resolves build intent, prepares locked lanes, runs the build
  scheduler, executes planned actions, records cache generations, and returns
  `Build_result.t`.
- `riot-planner` produces target-aware package, module, and action plans.
- `riot-store` owns artifact storage, cache lookup, generation indexing, and
  GC policy.
- `riot-toolchain` owns compiler and cross-toolchain discovery.

The separate `riot-executor` package should be folded into `riot-build`.
Execution is not an independent public domain in the current system:
`riot-build` already owns lanes, locks, result aggregation, cache generation
recording, and public build events, while `riot-executor` exposes package
results and telemetry that `riot-build` and `riot-cli` reach through directly.
That makes `riot-executor` a leaked implementation boundary, not a reusable
library boundary.

Folding it does not mean flattening the code into one module. `riot-build`
should still keep internal modules for scheduler state, action execution,
sandbox preparation, diagnostic rewriting, package result aggregation, and
telemetry. The difference is that those modules become private implementation
details behind `Riot_build.build` and `Riot_build.Event`, so build invariants
can be expressed across request resolution, locking, planning, execution, and
finalization without exporting half-built executor types.

`riot-build` should not own:

- workspace scanning
- dependency solving
- registry or source package loading
- package creation
- executable lookup
- test or benchmark suite policy
- format or fix command policy

Those responsibilities belong to their command packages or domain packages.

### Public build surface

The public `Riot_build` facade should stay small:

```ocaml
module Request
module Event
module Build_result
module Build_lock

type error

val build :
  ?on_event:(Event.t -> unit) ->
  Request.t ->
  (Build_result.t, error) result
```

`Request.t` should contain only typed edge data:

- prepared `Workspace.t`
- requested packages as `Package_name.t list`
- requested targets as `Target.request`
- profile
- scope
- optional requested parallelism

It should not contain strings that still need parsing. String targets and
package names should be parsed at the CLI or command-package edge.

Requested parallelism is user intent, not the machine limit. If the request
does not specify it, Riot should default the request to the available
parallelism discovered from the host. If the request asks for more workers than
the host exposes, Riot clamps the worker budget to the available parallelism.
Zero or negative worker counts are invalid requests and must fail fast with
`jobs must be >= 1`. The final clamped value is the only parallelism value
stored in the build context.

### Build context

At the start of a build, `riot-build` creates one immutable context:

```ocaml
type t = {
  session_id : Session_id.t;
  workspace : Workspace.t;
  profile : Profile.t;
  host : Target.t;
  toolchain_config : Toolchain_config.t;
  parallelism : int;
  on_event : Event.t -> unit;
}
```

This eliminates repeated calls for the same session id, host triple, workspace
root, and concurrency budget. It also gives every event one source of timing
and session identity.

Context creation is the only place that combines requested and available
parallelism, and it must reject invalid values:

```ocaml
let requested = Request.parallelism request |> Option.unwrap_or ~default:available in
let parallelism = Int.min available requested
```

After that point, scheduler code uses `Build_context.parallelism`. It should
not independently read host parallelism, and it should not need to know what
the user originally requested.

### Resolved build intent

Request resolution should produce an opaque resolved value:

```ocaml
module Resolved_build : sig
  type t

  val resolve :
    Build_context.t ->
    Request.t ->
    (t, error) result
end
```

`Resolved_build.t` means:

- package names exist in the prepared workspace
- target request matched at least one target
- duplicate targets have been removed
- scope and profile are fixed

Functions after this point should not re-check whether a package exists or
whether a target pattern matched. If they need those facts, they should require
`Resolved_build.t`.

### Lanes and locks

A lane is one profile/target compilation environment.

```ocaml
module Lane : sig
  type unresolved
  type locked

  type 'stage t

  val prepare :
    Build_context.t ->
    Resolved_build.t ->
    target:Target.t ->
    (locked t, error) result
end
```

The first non-trivial action for a lane is acquiring its build lock. A lane
without the lock is not allowed to plan, execute, write sandbox state, promote
outputs, or record cache generation data.

The locked lane should contain:

- the target triple
- the profile name
- `Build_ctx.t`
- compilation mode
- initialized toolchain for that target
- store lane
- lock token

The exact phantom-type shape is not mandatory if opaque modules give the same
compile-time safety, but the API should make it impossible to call lane
execution functions with an unlocked target.

For multi-target builds, duplicate target triples are removed before locks are
created. Lanes are independent after locking. A worker should never need to hold
more than one lane lock at a time.

### Scheduler work items

The scheduler should own one queue of build work:

```ocaml
type work_item =
  | PlanPackage of {
      lane : Lane.locked Lane.t;
      package_key : Package.key;
    }
  | ExecuteAction of {
      lane : Lane.locked Lane.t;
      package_key : Package.key;
      action : Riot_planner.Action_node.t;
    }
  | FinalizePackage of {
      lane : Lane.locked Lane.t;
      package_key : Package.key;
    }
```

This is intentionally one queue, not a package queue plus an action queue plus
a target loop. Readiness determines what can run. The scheduler owns:

- ready work
- waiting work
- active work
- completed package results
- completed action results
- failed dependencies
- package/lane state

Workers consume ready work under one concurrency budget:

```ocaml
Scheduler.run :
  Build_context.t ->
  lanes:Lane.locked Lane.t list ->
  graph:Riot_planner.Package_graph.t ->
  (Lane_result.t list, error) result
```

If `Build_context.parallelism = 1`, there is one worker and the scheduler
behaves serially. If it is `10`, there are at most ten workers across all
planning and execution work. Riot should not independently create ten planning
workers and ten action workers for the same build.

### Package planning work

`PlanPackage` checks package dependency readiness first.

If dependencies failed, the package becomes skipped.

If dependencies are incomplete, the work item moves to a waiting set keyed by
the dependency packages.

If dependencies are satisfied, the worker calls the planner with the lane's
`Build_ctx.t`:

```ocaml
Riot_planner.plan_package_with_graph
  ~workspace
  ~toolchain
  ~store
  ~package_graph
  ~package_key
  ~package
  ~build_ctx:lane.build_ctx
```

The planner result can be:

- cached package artifact
- planned module/action graph
- missing dependencies
- failed dependencies
- planning error

On a cached package, the scheduler records the package result, emits cache-hit
events, updates dependents, and does not enqueue actions.

On a planned package, the scheduler stages the package runtime and enqueues
ready action nodes as `ExecuteAction` work items.

Planning remains target-aware even if the first implementation still produces
the same graph for every target. The API should assume the graph can differ per
lane.

### Action execution work

`ExecuteAction` checks the action cache before doing process work.

The cache check can happen in either place:

- before enqueuing the action, which avoids queueing known cached actions
- inside the worker, which keeps the queue simpler

The first implementation should prefer the simpler worker check unless
measurement shows the queue overhead is meaningful. The important invariant is
that the action result is reported through the same scheduler state whether it
was cached or freshly executed.

On completion, the scheduler:

1. records the action result for the package/lane
2. emits action events
3. wakes dependent actions in the same package
4. finalizes the package when all actions are accounted for
5. wakes package dependents when the package result is available

This preserves action-level caching from RFD0012 while removing the separate
package-planning and action-execution ownership split.

### Package finalization

`FinalizePackage` creates the package-level result once all actions for a
package/lane are complete.

Finalization should:

- collect warnings
- compute exported artifacts
- save or materialize the package artifact through `riot-store`
- mark the package graph node as cached or fresh
- emit package completion events
- clean sandbox state

The final result is keyed by package name and lane. `Build_result.t` can still
present the user-facing package view, but internally the scheduler must not
forget the lane that produced the artifact.

### Target-aware planning and conditional compilation

This RFD does not define conditional-compilation syntax. It defines the build
shape needed to support it later.

The rule is:

> planning happens inside a target/profile lane and receives that lane's
> `Build_ctx.t`.

That means future features can make these target-dependent:

- source inclusion
- dependency inclusion
- foreign dependency configuration
- compiler flags
- generated files
- module graph shape
- action graph shape

The scheduler should never assume that one package plan can be reused across
targets unless a planner-level cache proves the target-aware input hash is
identical.

### Cache and store interaction

The hot path should remain content-addressed:

- package planning caches are keyed by package input hash plus relevant lane
  inputs
- action caches are keyed by action hashes
- package export manifests reference action hashes
- successful builds hand lane closure data to `riot-store`
- `riot-store` decides whether generation bookkeeping can be skipped cheaply

The scheduler should not read all generation manifests to decide whether a
fully cached build is known. That optimization belongs in `riot-store`, which
already owns cache state and generation indexing.

### Events

The build event stream should be direct and typed. Work should emit events from
the place where it happens:

- request resolution emits target-resolution events
- lane preparation emits toolchain and lock events
- package planning emits package-planning events
- action execution emits action events
- finalization emits package result events
- cache generation recording emits cache generation events

`riot-build` should not need to translate from an internal response protocol
back into its own public build events. If executor telemetry remains a lower
level event type, the bridge should be explicit and lossless for JSON mode.
Dropping action or planning events silently should not be the default behavior.

### Failure and cancellation policy

The initial scheduler should preserve current build semantics:

- failed packages cause dependent packages to skip
- independent packages may continue if partial failure mode allows it
- normal `riot build` fails if any required package fails
- package results include skipped dependents with reasons

The scheduler state should make fail-fast and cancellation policy explicit, but
this RFD does not require introducing user-facing cancellation controls.

## Drawbacks
[drawbacks]: #drawbacks

This is a real rewrite, not a cleanup pass.

The main drawback is implementation risk. Build orchestration is central to
Riot, and concurrency bugs in this layer can be subtle. The rewrite would fold
`riot-executor` into `riot-build`, touch parts of `riot-planner`, and update
test fixtures that assert event ordering.

The second drawback is conceptual weight. A unified work scheduler is a
stronger abstraction than a simple "plan packages, then execute actions" flow.
It needs good tests, good names, and clear invariants or it can become harder
to debug than the current split.

The third drawback is event churn. Better event visibility may change JSON
streams that downstream tools consume. Riot should treat event shape changes as
intentional API work, not incidental logging.

The fourth drawback is nondeterminism. More concurrency across packages,
actions, and targets can produce different event interleavings. The result
must stay deterministic where it matters, especially cache keys, artifact
paths, final package status, and error summaries.

## Rationale and alternatives
[rationale-and-alternatives]: #rationale-and-alternatives

### Do nothing

Doing nothing keeps the current split:

- `riot-build` has session-loop baggage
- target lanes are orchestrated above the executor
- package planning and action execution remain separate concepts
- JSON output can have quiet gaps where the build is doing real work

This is the cheapest option now, but it keeps taxing every future build-system
change.

### Rename the transport-shaped modules

Renaming `client`, `server`, and `protocol` modules reduces confusion, but it
does not change the architecture. It still leaves a request/response-shaped
local system and does not improve type-level build invariants, scheduling, or
event visibility.

This can be useful as a tiny readability improvement, but it is not the design
change this RFD is about.

### Direct in-process build, but keep phase separation

Another option is to remove the local request/response loop and keep the
current planner/executor split mostly unchanged.

That would solve some code complexity in `riot-build`, but it would not solve
the larger scheduling issue. Riot would still have package planning feeding an
action queue from the side, and multi-target builds would still tend to be
organized as lane loops rather than scheduler work.

### Keep `riot-executor` as a separate package

Keeping `riot-executor` separate preserves a familiar package split: planner
plans, executor executes, build orchestrates.

The problem is that this split is already false in practice. `riot-build`
depends on executor package-result types in its public API, translates
executor telemetry into build events, owns build locks and generation
accounting around executor results, and chooses target lanes above the
executor. `riot-cli` also reaches directly into executor result/telemetry
types for formatting. A separate package boundary therefore does not protect a
clean abstraction; it forces the abstraction to leak.

The better boundary is internal: keep action execution as an internal
`riot-build` module, but make `Riot_build` the only package-level build API.

### Plan everything first, then execute

Planning the full workspace before execution is simpler to reason about. It
also gives a complete graph before workers start.

The cost is late feedback and reduced overlap. Riot would keep paying all
planning cost before the first compiler action can fail. It also pushes the
system toward target-independent planning, which is the wrong default for
conditional compilation.

### Extend only the existing action queue

The existing `Action_queue` is useful and should inform the implementation.
But extending only that queue keeps package planning as a separate state
machine. The proposal here is to make package planning itself scheduler work,
so the same concurrency budget and readiness logic applies to packages,
actions, and lanes.

### Proposed design

The proposed design is the best fit because it addresses the structural causes
instead of one symptom:

- it deletes the local protocol boundary
- it deletes the leaked executor package boundary
- it makes the build order visible in typed values
- it starts work as soon as dependencies allow it
- it shares one concurrency budget
- it models target-specific planning from the beginning
- it keeps action-level caching as the durable execution model

## Prior art
[prior-art]: #prior-art

RFD0003 documented Riot's previous local-session build shape and explicitly
noted that the "server" was an in-process orchestration boundary rather than a
long-lived daemon. This RFD takes the next step: if it is not a daemon and not
an external peer, Riot should not keep a protocol-shaped design internally.

RFD0012 proposed action-level caching and workspace-level action scheduling.
That RFD remains the right cache direction. This RFD extends the same idea by
moving package planning into the scheduler model instead of treating it as a
separate phase that feeds action execution.

RFD0032 defined successful build generations and cache GC. This RFD keeps that
responsibility in `riot-store`; the scheduler produces lane closure data, and
the store owns cheap state-based decisions about whether generation accounting
can be skipped.

Build systems such as Ninja, Bazel, Buck, and Dune all point toward the same
lesson: the scheduler should own readiness, cache checks, and worker budgeting
around graph work. Riot does not need to copy their architectures wholesale,
but it should adopt the principle that ready work should start as soon as the
build graph permits it.

Riot's actor runtime is also relevant prior art. It already provides a mental
model for bounded worker pools and message-driven work, but `riot-build`
should not expose an accidental protocol just because workers internally
communicate. The public and package-internal design should be typed build
stages, not actor messages.

## Unresolved questions
[unresolved-questions]: #unresolved-questions

- Should package planning cache hits be represented as completed package work
  or as a separate cache-hit work item?
- Should action cache checks happen before enqueuing actions or inside worker
  execution?
- What fairness policy should prevent one target lane or one large package
  from dominating all workers?
- What is the exact event compatibility policy for richer JSON output?
- How should lane locks be acquired for many targets if one target lane is
  already locked by another process?
- What is the precise target-aware package input hash once conditional
  compilation exists?
- Should the first implementation preserve current "continue independent
  packages" behavior exactly, or introduce an explicit fail-fast mode?

## Future possibilities
[future-possibilities]: #future-possibilities

Target-aware planning unlocks conditional compilation without another build
runtime redesign. Future syntax can affect source selection, dependencies, and
foreign configuration because the scheduler already plans per lane.

A unified work scheduler also creates a cleaner path toward remote caching and
remote execution. If every executable unit is already a typed work item with a
stable content hash and lane configuration, remote execution can be introduced
as a worker backend rather than a second scheduler.

Better event structure can improve editor and CI integrations. JSON output can
show whether Riot is resolving targets, waiting for locks, planning packages,
checking cache, executing actions, finalizing packages, or recording cache
generations.

The same scheduler model could later support cancellation, progress snapshots,
interactive diagnostics, and build graph inspection. Those are out of scope for
this RFD, but the proposed architecture gives them a clearer place to attach.

Finally, once package planning and action execution share one queue, Riot can
measure actual scheduler utilization. That gives future performance work a
better target than guessing whether time was spent in planning, cache lookup,
execution, or coordination.
