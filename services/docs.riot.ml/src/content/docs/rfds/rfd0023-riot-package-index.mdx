## Summary
[summary]: #summary

This RFD documents Riot's sparse package index as a current-system snapshot.
The index is a Cargo-style sharded JSON read model rooted at
`https://cdn.pkgs.ml/index/v1/` and exists to make named package installs fast,
cache-friendly, and independent from the registry control plane at read time.

- the index is written synchronously during publish and then served publicly
  from `cdn.pkgs.ml`
- it contains only explicitly published, named public packages
- package documents are sharded so a named install needs one config fetch plus
  one small package fetch in the common case
- clients solve versions locally after reading the index document
- the index is an install read model, not an ownership, authentication, or
  source-materialization service

## Motivation
[motivation]: #motivation

The registry solves publication and provenance, but named package installation
needs a much cheaper and more CDN-friendly hot path.

If the install path had to stay on the registry control plane, Riot would keep
paying for the wrong shape of system:

- control-plane availability would matter too much for ordinary installs
- package-name lookup would be harder to cache aggressively
- install UX would mix publication concerns with read-heavy package resolution

The desired named-install workflow is much narrower:

```text
riot add kernel
```

should usually mean:

1. fetch sparse index config once
2. compute one shard path
3. fetch one package document
4. solve locally in Riot
5. download immutable artifacts by SHA

That is why the current index is static, sharded, and deliberately independent
from GitHub and registry-only control-plane logic at read time.

## Guide-level explanation
[guide-level-explanation]: #guide-level-explanation

The easiest way to understand the index is through a named install.

Suppose a contributor runs:

```text
riot add kernel
```

The current mental model is:

- the registry made `kernel` publishable
- the sparse index tells the client what published releases exist
- Riot solves locally and then downloads the chosen immutable artifact

### Ownership

The index does not own:

- source materialization
- authentication
- package-name claims
- publish authorization
- search
- docs

Those are registry concerns or downstream concerns.

The index owns only the install read model for named packages.

### What gets indexed

A release is indexable when:

1. it was explicitly published
2. its package name is claimed
3. its version is valid semver
4. the package is public

The index does not include:

- unpublished source materializations
- mutable refs like `main`
- raw SHAs without a named release

### Sparse layout

The implemented shard layout matches the Cargo sparse index strategy with JSON
files:

```text
index/v1/
  config.json
  1/x.json
  2/io.json
  3/m/mcp.json
  ke/rn/kernel.json
  mi/nt/minttea.json
```

### Package document model

Each package shard contains one package document with:

- package name
- latest release version
- updated timestamp
- all published releases for that package

Each release entry includes:

- version
- published timestamp
- artifact SHA-256
- description
- license
- homepage
- repository
- optional provenance metadata
- repository
- root module
- categories
- keywords
- manifest key
- source key
- dependency metadata

This lets `riot` solve locally without consulting the registry Worker on the
install hot path.

## Publish-time behavior
[reference-level-explanation]: #reference-level-explanation

On successful publish, the registry now performs index work synchronously:

1. read the current package document from R2, if any
2. upsert the release into the package document
3. semver-sort releases
4. recompute `latest`
5. write the package shard back to R2 if it changed
6. emit `package.searchable`
7. emit `package.indexed`

Because this work happens inline with publish, the user-visible behavior is:

- publish succeeds
- the package is already visible in the sparse index
- `riot add <package>` can use it immediately

## Root config

The root config document at `v1/index/config.json` tells clients:

- index kind
- shard strategy
- index base URL
- artifact base URL

This gives `riot` a single bootstrap file to cache and use for all future
named-package installs.

## Client contract for future `riot add`

The index is the main contract for named installs.

`riot add <package>` should:

1. fetch `config.json`
2. compute the shard path for the package name
3. fetch the package document
4. select a compatible version locally
5. fetch the immutable package artifact referenced by the chosen release

The index deliberately does not choose versions for the client.
That belongs in Riot's package-management layer.
