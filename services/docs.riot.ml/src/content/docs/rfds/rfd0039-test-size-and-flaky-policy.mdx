- Start Date: `2026-04-07`
- Status: `implemented`

## Summary
[summary]: #summary

This RFD introduces shared test metadata and repository-local execution policy
for Riot test suites. It is a test-runner and workflow RFD, not a build-graph
change and not a project-manifest change.

- `Std.Test` test cases gain `size` and `reliability` metadata.
- `reliability` is explicit and typed: `Stable | Flaky of { retry_attempts:
  int }`.
- `small` is the default size bucket; `large` opt-outs are the slow tests that
  repositories can skip in pre-commit and other fast feedback loops.
- `.riot/config.toml` grows a `[riot.test]` section for repository-shared test
  policy such as `small_test_timeout` and `flaky_max_retries`.
- `riot test --small`, `--large`, and `--flaky` partition the case set through
  the shared `run-tests` contract instead of inventing a second filtering layer
  in the CLI.
- Each test attempt runs in its own child actor so the runner can monitor it,
  time out `small` tests, and kill the attempt actor before retrying a flaky
  case.

## Motivation
[motivation]: #motivation

Riot currently treats all test cases as identical once a suite binary starts
running.

That creates two concrete problems in day-to-day contributor workflows:

1. Repositories cannot cleanly separate fast pre-commit test cases from known
   slow tests.
2. A small number of flaky timer- and receive-driven tests can fail a
   pre-commit hook even when rerunning them a few times would usually pass.

Today Riot pays for that in three ways:

- pre-commit hooks either run everything, including slow tests, or maintain
  package-local ad hoc allowlists
- flaky tests have no shared retry contract, so every failure is terminal even
  when the test is known to be operationally noisy rather than semantically
  broken
- repository-shared policy has no way to say "small tests should fail fast
  here, but large tests are allowed to run"

These costs are structural because Riot currently has no shared vocabulary for:

- expected test runtime class
- flaky retry eligibility
- repository-local test execution policy

This RFD fixes that by giving Riot one shared system:

- `Std.Test` owns the test metadata and suite-runner behavior
- `.riot/config.toml` owns repository-local test policy
- `riot test` owns the user-facing filtering workflow

The motivating workflows are straightforward:

- a pre-commit hook can run `riot test --small` and skip known slow tests
- a periodic or CI job can run `riot test --large`
- a debugging pass can run `riot test --flaky`
- flaky tests can retry a bounded number of times without every package
  inventing its own retry loop

## Guide-level explanation
[guide-level-explanation]: #guide-level-explanation

Contributors should think about Riot tests as having two orthogonal pieces of
metadata:

1. runtime size: `small` or `large`
2. reliability: `stable` or `flaky`

The default case is a stable small test:

```ocaml
let tests = [
  Test.case "parser roundtrip" (fun _ctx -> Ok ());
]
```

Slow tests opt into the large bucket:

```ocaml
let tests = [
  Test.case ~size:Test.Large "integration handshake" (fun _ctx -> Ok ());
]
```

Known noisy tests opt into retry handling:

```ocaml
let tests = [
  Test.case ~reliability:Test.(Flaky { retry_attempts = 3 }) "timer race" (fun _ctx -> Ok ());
]
```

Repository policy lives in `.riot/config.toml`:

```toml
[riot.test]
small_test_timeout = "500ms"
flaky_max_retries = 3
```

That means a repository can teach contributors one simple workflow:

- `riot test --small` for fast feedback
- `riot test --large` for slow suites
- `riot test --flaky` for noisy cases

Combined filters are intersectional:

- `riot test --small --flaky` runs only flaky small tests
- `riot test --large --flaky` runs only flaky large tests

`--small` and `--large` are mutually exclusive.

From the suite-binary side, the same partitioning lives on `Std.Test.Cli`:

```text
run-tests --small
run-tests --large
run-tests --flaky
run-tests --small --flaky
```

Small-test timeout is enforced inside the suite runner. Large tests have no size
timeout. Flaky tests rerun on failure or timeout until they either pass or hit
their retry budget, optionally capped by repository policy.

This rollout keeps the isolation model lightweight: timed and retried attempts
run in isolated child actors inside the suite process rather than full OS
subprocesses. That is enough for fast feedback workflows and for the current
class of timer- and receive-driven flaky tests while keeping the contract
portable across Riot test binaries.

## Reference-level explanation
[reference-level-explanation]: #reference-level-explanation

## Metadata

`Std.Test.test_case` gains:

- `size`
- `reliability`

The public size vocabulary is:

- `Small`
- `Large`

Unannotated tests default to `Small`.

The public reliability vocabulary is:

- `Stable`
- `Flaky of { retry_attempts: int }`

Unannotated tests default to `Stable`.

## Runner filtering

`Std.Test.Cli run-tests` accepts:

- `--small`
- `--large`
- `--flaky`
- `--small-timeout-ms <int>`
- `--flaky-max-retries <int>`

Query substring matching remains separate from metadata filters.

## Runner execution model

Each test attempt runs in a fresh child actor.

The parent runner actor:

- monitors the child actor
- applies the `small_test_timeout` budget when the test is `Small`
- kills the child actor on timeout
- decides whether to retry based on the test's `reliability` metadata and the
  configured retry cap

This keeps timeout and retry handling inside the shared runner contract without
requiring one OS subprocess per test case.

## Repository policy

`.riot/config.toml` gains:

```toml
[riot.test]
small_test_timeout = "500ms"
flaky_max_retries = 3
```

This section is repository-local operational policy and therefore belongs in
`Workspace_operational_config`, not in `riot.toml` and not in
`~/.riot/config.toml`.

`flaky_max_retries` is a repository-local cap on retry behavior. A flaky test
still declares its own retry budget through `Flaky { retry_attempts }`; the
effective retry count is the smaller of the test-local value and the
repository-local cap when the cap is non-zero.

## Result shape

Structured suite JSON now carries per-test:

- `size`
- `reliability`
- `retry_attempts` when the test is flaky
- `attempts`
- `duration_us`
- timeout-aware status information

`riot-build` preserves those fields in aggregated test events so `riot-cli` can
render retries and timeout failures without scraping pretty output.

## Drawbacks
[drawbacks]: #drawbacks

- Riot gains more test-runner surface area and more policy to explain.
- Small-test timeout is actor-isolated and cooperative within the suite
  process, not true subprocess isolation.
- Defaulting unannotated tests to `small` means some existing slow tests must
  be reclassified as `large` to keep fast workflows healthy.

## Rationale and alternatives
[rationale-and-alternatives]: #rationale-and-alternatives

### Why default to `small`?

Because the motivating workflow is fast feedback. Requiring every fast test to
be explicitly annotated would make the feature expensive to adopt and would not
help existing repositories until a large annotation pass finished.

### Why not read `.riot/config.toml` directly from `Std.Test`?

Because `Std.Test` should own the portable runner contract, not Riot-specific
repository config discovery. The CLI and build layers load repository policy
and forward it explicitly.

### Why child actors instead of one subprocess per test case?

Subprocesses would provide stronger timeout isolation, but they are a much
larger runtime and process-management change. Child actors let the shared
runner keep timeout and retry policy inside `Std.Test`, which is enough for the
current flaky-test motivation and leaves stronger isolation as follow-up work
if the lightweight model proves insufficient.

## Prior art
[prior-art]: #prior-art

- Test metadata and timeout classes in other test runners
  - The important lesson is the split between case metadata and repository
    policy, not any exact flag spelling.
- Riot's existing shared `Std.Test.Cli` contract
  - This RFD extends that contract instead of introducing package-local test
    CLIs or pre-commit-only wrappers.
- `RFD0038`
  - Repository-local operational behavior already has a home in
    `.riot/config.toml`; test policy follows that boundary.

## Unresolved questions
[unresolved-questions]: #unresolved-questions

- Whether Riot should eventually promote timeout isolation from child actors to
  per-test subprocesses.
- Whether future workflows need a third size class beyond `small` and `large`.

## Future possibilities
[future-possibilities]: #future-possibilities

- process-level per-test isolation for stronger timeout guarantees
- repository-local defaults for more test-runner behavior
- richer flaky reporting such as first-failure retention across retries
