- Feature Name: `test_coverage_and_lcov_reporting`
- Start Date: `2026-04-07`
- Status: `presented`

## Summary
[summary]: #summary

This RFD proposes built-in source coverage for `riot test`, with Riot
instrumenting workspace `src/` modules in the `test` build lane and emitting
an LCOV report at `coverage/lcov.info`.

* it measures first-party workspace sources only, so contributors can see what
  their own tests exercise without mixing in third-party dependencies

* it performs coverage instrumentation in one CST traversal per file, using
  that pass to inject probe hits and record the source-oriented probe map for
  later reporting

* it collects coverage during a normal `riot test` invocation, flushes raw data
  per suite, and performs an explicit merge/report step at the end

* it snapshots source text for reporting, so coverage remains tied to the code
  that was actually tested even if files change later

* it standardizes on LCOV as the first external artifact so CI pipelines can
  plug into existing coverage tools without Riot needing provider-specific
  upload integrations

## Motivation
[motivation]: #motivation

Riot currently gives contributors no built-in answer to a basic testing
question:

- how much of my code do these tests actually exercise?

That gap is increasingly expensive because Riot is now a large OCaml codebase.
At the time of this RFD, Riot is sitting at roughly `313,564` lines of OCaml.
A passing test run can tell us that certain assertions succeeded, but it still
cannot tell us what parts of the workspace those tests actually touched.

That hurts Riot in exactly the workflows where confidence matters:

1. **large refactors**: a green test run is not enough if the changed files
   were barely exercised in the first place

2. **AI-driven refactors**: automated edits can move quickly, but Riot still
   needs a way to show which files remain weakly tested after those edits land

3. **feature work**: when adding behavior, contributors need to know where the
   weak spots are so they can add tests that genuinely raise confidence

4. **CI reporting**: pipelines can prove that tests passed, but Riot has no
   built-in standard artifact showing how much first-party code those tests
   covered

Today Riot pays for that in several ways at once:

- contributors cannot reliably find untested or weakly tested areas of the
  workspace
- "all tests passed" is easy to overread as "this code is well exercised"
- repositories that want coverage in CI must reach outside Riot and duplicate
  too much of Riot's package and runner knowledge

That last cost is structural, not incidental. Riot already owns:

- workspace planning
- source ownership
- package discovery
- test suite discovery
- suite execution
- structured test result collection

If coverage lives entirely outside Riot, an external tool has to partially
reconstruct all of that just to answer one basic question:

- which first-party workspace sources were exercised by this test run?

That duplication is the wrong shape. Coverage belongs next to Riot's existing
test runner because Riot is already the system that knows:

- what counts as workspace code
- what should and should not be instrumented
- how suites are built
- how suites are executed

The primary users for this feature are:

- Riot users who want to find weak spots in their testing story
- CI pipelines that want a standard artifact they can upload to existing
  coverage providers

The minimum successful first version is intentionally small:

- `riot test --coverage` works
- Riot emits `coverage/lcov.info`
- contributors can point at poorly tested files and write better tests for them

## Guide-level explanation
[guide-level-explanation]: #guide-level-explanation

We'll explain this with two examples, a local refactor and a CI upload.

## Local refactor

Suppose a contributor is about to make a broad change across several modules in
the workspace and wants to know whether the existing tests really exercise that
area.

Today the workflow looks like:

```text
riot test
```

and the answer Riot gives back is only:

- which suites ran
- which cases passed or failed

That is useful, but incomplete. A green run still leaves the contributor
guessing:

- which files were barely touched
- whether the changed modules are meaningfully exercised
- where to add tests before or after the refactor

With this proposal, the workflow becomes:

```text
riot test --coverage
```

Riot then:

1. builds the workspace in the `test` lane
2. instruments workspace `src/` modules during that build
3. runs the normal test suites
4. collects coverage hits while those suites execute
5. writes a final LCOV report to:

```text
coverage/lcov.info
```

Now the contributor has a concrete artifact they can inspect with existing
coverage tooling, and Riot can answer a much better question:

- what parts of my workspace code were actually exercised by this test run?

That changes the shape of the work:

- before a refactor, Riot can show which files are weakly covered
- after a refactor, Riot can show whether the changed areas are exercised
- when confidence is low, contributors know where to add tests next

## CI upload

The second workflow is CI.

Today a Riot repository can run tests in CI, but there is no built-in standard
coverage artifact for existing reporting services to consume.

With this proposal, the CI shape becomes:

```yaml
- run: riot test --coverage
- uses: codecov/codecov-action@v4
  with:
    files: coverage/lcov.info
```

That works because Riot standardizes one provider-neutral output:

```text
coverage/lcov.info
```

Riot itself does not need to learn provider-specific APIs, tokens, or PR
annotation formats. It just needs to emit a durable, standard report.

## Mental model

Contributors should think about coverage as a normal `riot test` execution mode
with one extra build behavior:

- Riot builds tests in the `test` lane
- when coverage is enabled, Riot instruments workspace `src/` files in that
  lane
- suites run normally
- coverage is merged after the suites finish
- the final durable output is `coverage/lcov.info`

The important thing Riot is adding is not just another report file. It is a
shared answer to a workflow question contributors already have:

```text
how much of my code did these tests really exercise?
```

## Reference-level explanation
[reference-level-explanation]: #reference-level-explanation

## Build lane

`riot test --coverage` builds and executes in the `test` lane.

This RFD intentionally does not support `riot test --release`. The v1 contract
is:

- `riot test` builds in `_build/test/<target>/...`
- `riot test --coverage` uses the same `test` lane with coverage
  instrumentation enabled

This keeps the first rollout simple and aligns coverage with the future desire
for test-specific conditional compilation.

## Coverage scope

Coverage applies to workspace `src/` modules only.

That means:

- instrument workspace-owned first-party source files
- do not instrument third-party dependencies
- do not instrument `tests/`
- do not instrument generated support that does not belong to workspace `src/`

The final report should still cover all workspace sources that participate in
the tested workspace closure, not only the package where the selected test
suite lives.

## CLI surface

`riot test` gains:

```text
--coverage
--coverage-reporter=<val>
--coverage-dir=<val>
```

For v1:

- `--coverage` enables instrumentation and final report generation
- `--coverage-reporter` defaults to `text,lcov`
- `--coverage-reporter=lcov` or `text,lcov` enables LCOV output
- `--coverage-dir` defaults to `coverage`

This RFD standardizes `coverage/lcov.info` as the initial durable artifact.

`riot test --json` should emit one final coverage event that points at the
generated LCOV file when coverage is enabled.

## Instrumentation model

Coverage instrumentation happens during the test build through a source rewrite
pass over each instrumented file.

The key design constraint is:

- one instrumentation traversal per CST per file

During that traversal Riot should:

1. select probe points
2. assign stable probe ids within the file
3. rewrite the relevant bodies to call the coverage runtime
4. build the source-oriented probe map for the file

The runtime shape is:

```ocaml
let __coverage =
  Std.Test.Coverage.register_module
    ~file:"packages/foo/src/bar.ml"
    ~source_hash:"..."
    ~probes:[| ... |]
```

and then probe hits look like:

```ocaml
Std.Test.Coverage.hit __coverage <probe_id>
```

Probe metadata is source-oriented and recorded once at module registration
time. Probe hits carry only the module handle and probe id.

This avoids paying source-location allocation costs on every hit.

## Probe policy

The v1 instrumentation policy is intentionally narrow.

Instrument:

- function bodies
- `if` / `else if` / `else` branch bodies
- `match` arm bodies
- `try ... with` handler bodies

Do not instrument:

- every nested expression
- type declarations
- signatures
- patterns
- test modules
- third-party dependencies

This keeps the rewrite predictable, the runtime overhead modest, and the
resulting LCOV mapping understandable.

## Probe map and source snapshots

Each instrumented file produces a probe map that includes:

- workspace-relative source path
- source hash
- the original source snapshot
- the list of probes with source spans

The important rule is:

- the final report is generated from the original source snapshot and probe map
- Riot does not attempt to reconstruct coverage from rewritten source text

This guarantees stable reporting even if the workspace source file changes
after the test run finishes.

## Runtime collection

Coverage state lives in memory for the duration of a `riot test` invocation.

Per-suite behavior:

- suites execute normally
- coverage hits accumulate in process-local runtime state
- when a suite completes, Riot flushes that suite's raw coverage data to disk

After suite execution completes, Riot performs an explicit final merge/report
generation step.

This RFD does not require the raw per-suite on-disk format to be part of the
public contract. It only requires that the merged final output be stable and
that the final report can be generated from the flushed suite data plus the
captured source snapshots.

## Final outputs

The final merged report lives under the workspace root:

```text
coverage/
```

For v1, Riot overwrites that directory's coverage outputs for each run.

The main external artifact is:

```text
coverage/lcov.info
```

Human-readable text reporting may also be printed during the command, but the
LCOV file is the durable provider-facing contract for this RFD.

## Failure behavior

Coverage generation is best-effort relative to test execution.

If some suites fail but coverage data exists, Riot should still:

- merge the available coverage data
- write `coverage/lcov.info`
- emit the final coverage event in JSON mode

The command should still exit non-zero for test failure, but coverage should
not disappear just because some test cases failed.

If no coverage data exists, empty output is acceptable. Riot should not turn
that into a hard command error.

## Drawbacks
[drawbacks]: #drawbacks

- Riot gains a new instrumentation and reporting subsystem in a critical test
  workflow.
- Coverage builds are more expensive than normal test builds.
- Source rewriting increases implementation complexity in the test lane.
- LCOV is line-oriented, so Riot's richer internal probe model will be
  flattened to a simpler external format in v1.

## Rationale and alternatives
[rationale-and-alternatives]: #rationale-and-alternatives

### Why build this into Riot instead of relying on external tooling?

Because Riot already owns the hard parts:

- workspace source ownership
- package planning
- suite discovery
- suite execution

External tooling would need to partially rebuild Riot's planner and runner
model just to answer "what first-party code did these suites exercise?"

### Why LCOV first?

Because LCOV is already widely accepted by existing coverage tooling and CI
providers. It gives Riot one standard export format without coupling Riot to a
provider-specific upload workflow.

### Why not instrument tests too?

Because the immediate goal is to show how much of the workspace's first-party
source code is exercised by tests.

Covering `tests/` would add noise to the first report, complicate the story,
and does not directly answer the motivating question.

### Why not support `riot test --release` in v1?

Because the first rollout is easier to reason about if the test command always
uses the dedicated `test` lane. That also leaves room for future test-specific
conditional compilation without immediately defining a larger profile matrix.

## Prior art
[prior-art]: #prior-art

- Cargo's dedicated `test` profile and separate test execution model
  - Riot follows the same broad lesson that tests deserve a dedicated lane.
- Bun's `bun test --coverage` CLI shape
  - Riot borrows the general user-facing coverage flag model.
- LCOV as a common interchange format
  - Riot uses the existing ecosystem contract instead of inventing a
    provider-specific uploader.

## Unresolved questions
[unresolved-questions]: #unresolved-questions

- What exact internal raw file format should Riot use for flushed per-suite
  coverage data?
- How should the implementation boundary settle between `riot-build` and a
  future `riot-test` package that centralizes test command domain logic?
- Should a future rollout add richer Riot-native JSON coverage output beyond
  the final LCOV path event?

## Future possibilities
[future-possibilities]: #future-possibilities

- add richer text summaries and file-level weak-coverage reporting
- add branch-aware reporting on top of the same probe metadata
- integrate future test-only conditional compilation on top of the dedicated
  `test` lane
- expose a Riot-native machine-readable coverage report format alongside LCOV
