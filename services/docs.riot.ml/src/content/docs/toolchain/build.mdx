`riot build` is the backbone of the Riot toolchain.

```sh
riot build
```

If that command does not feel predictable, the rest of the product will not feel predictable either.

## What users need from `riot build`

They need it to be:

- workspace-aware
- package-aware
- stable enough to trust
- aligned with later commands like `riot run`, `riot test`, and `riot doc`

This is why `riot build` is more than a compiler wrapper. It sits at the center of the toolchain story.

## The normal role of `riot build`

In day-to-day work, `riot build` usually means:

- confirm the workspace still compiles
- produce the artifacts needed for execution
- move from edit-time checking into an actual built state

It is the command people fall back to when they want stronger confidence than `riot check` alone.

## A useful mental model

Think of the command in three layers:

1. Riot figures out what the workspace means
2. Riot figures out what has to be built
3. Riot performs the build so later commands can rely on the result

That is what makes it feel integrated rather than ad hoc.

## Practical rhythm

A healthy development rhythm often looks like:

```sh
riot check
riot build
riot run <name>
riot test -p <pkg>
```

Not every edit needs a full build. But the build command is the point where the workspace has to stop
being hypothetical and become concrete.
