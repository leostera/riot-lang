## The adoption path

If you are still evaluating Riot, read these pages in order:

1. [Installation](/toolchain/installation)
2. [Quick Start](/toolchain/quickstart)
3. [Editor Setup](/toolchain/editor-setup)
4. [riot build](/toolchain/build)
5. [riot run](/toolchain/riot-run)

That is the shortest path from `what is this?` to `I can actually use this`.

## The daily loop

<div class="home-grid home-grid--cards">
  <a class="home-card" href="/toolchain/build/">
    <strong>riot build</strong>
    <span>Plan and build packages with the same dependency model the rest of Riot understands.</span>
  </a>
  <a class="home-card" href="/toolchain/check/">
    <strong>riot check</strong>
    <span>Run the fast validation loop when you want correctness signals without a full publish flow.</span>
  </a>
  <a class="home-card" href="/toolchain/fix/">
    <strong>riot fix</strong>
    <span>Apply syntax-directed fixes and keep codebases moving without manual cleanup passes.</span>
  </a>
  <a class="home-card" href="/toolchain/fmt/">
    <strong>riot fmt</strong>
    <span>Format the workspace with the same toolchain assumptions Riot uses everywhere else.</span>
  </a>
</div>

Once a workspace exists, the center of gravity usually becomes:

- [riot build](/toolchain/build)
- [riot check](/toolchain/check)
- [riot fix](/toolchain/fix)
- [riot fmt](/toolchain/fmt)
- [riot run](/toolchain/riot-run)

Those commands are not five unrelated features. Together they define the normal edit, validate, and
execute rhythm of a Riot workspace.
